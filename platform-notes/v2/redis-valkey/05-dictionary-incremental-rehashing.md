# Chapter 5: The Dictionary — Incremental Rehashing

> The hashtable is the single most important data structure in Redis. The
> top-level keyspace is a hashtable. Every database number is a hashtable.
> Every hash type with more than ~128 entries is a hashtable. Every sorted
> set's score-lookup table is a hashtable. Every cluster node's slot-to-key
> map is a hashtable. **Hashtables are everywhere.** And because Redis is
> single-threaded, the cost of growing a hashtable cannot be paid in one
> shot — that would freeze the server. So Redis pays it incrementally,
> across thousands of operations, and the algorithm that makes that work
> is one of the most beautiful things in the codebase.

---

## 5.1 What `dict` Actually Is

The `dict` is Redis's hash table. It uses **chained hashing with two tables**.
The two-table design is the foundation of incremental rehashing.

```c
// dict.h (Redis 7.x)
typedef struct dict {
    dictType *type;             // function pointers: hash, compare, free, ...
    dictEntry **ht_table[2];    // two hash tables (active + rehash target)
    unsigned long ht_used[2];   // entries in each table
    long rehashidx;             // -1 if not rehashing, else index of next bucket to migrate
    int16_t pauserehash;        // pause counter (>0 means rehash is paused)
    signed char ht_size_exp[2]; // size = 1 << ht_size_exp[i]
} dict;

typedef struct dictEntry {
    void *key;
    union {
        void *val;
        uint64_t u64;
        int64_t s64;
        double d;
    } v;
    struct dictEntry *next;     // chaining
} dictEntry;
```

Key features:

- **Two tables (`ht_table[0]` and `ht_table[1]`).** During steady-state, only
  `ht_table[0]` is live. During a rehash, both are live: lookups check `[0]`
  then `[1]`; inserts go straight to `[1]`.
- **Power-of-two size**, stored as exponent (`ht_size_exp = 4` → 16 buckets).
  Power-of-two means the bucket index is `hash & (size - 1)` — a single AND
  instruction, no expensive modulo.
- **Chained hashing**, not open addressing. Each bucket points to a linked list
  of entries that hash to it. Simpler to grow; no clustering issues.
- **`dictType`** is a vtable: hash function, key comparator, key/value
  destructor. The same `dict` code services the keyspace, the expires dict,
  the cluster's slot map, every Redis hash, every sorted set's score lookup.

```
   dict
   +---------+
   | type    |---> { hash=siphash, cmp=sdscmp, free=decrRefCount, ... }
   |---------|
   | ht[0]   |---> [ ptr, ptr, ptr, ptr, ... ]   (size = 1 << ht_size_exp[0])
   |         |          |
   |         |          v
   |         |       entry -> entry -> entry -> NULL  (collision chain)
   |---------|
   | ht[1]   |---> NULL  (only allocated during rehash)
   |---------|
   | rehashidx = -1   (or 0..ht_size[0]-1 during rehash)
   +---------+
```

## 5.2 The Hash Function: SipHash-2-4

Redis uses **SipHash-2-4** as its keyed hash function. SipHash is:

- **Fast**: ~1 GB/s on a modern CPU for short keys.
- **Cryptographically strong** for HashDoS resistance: an attacker cannot
  craft keys that all hash to the same bucket without knowing the seed.
- **Keyed**: the seed is randomly generated at server startup. Without
  knowing the seed, you can't predict bucket distribution.

```c
// siphash.c
uint64_t siphash(const uint8_t *in, const size_t inlen, const uint8_t *k);
```

Why this matters: before SipHash, Redis used MurmurHash2. MurmurHash is fast
but unkeyed and reversible — a 2017 paper showed how to construct
collision-bomb keys that turn O(1) lookup into O(N²) lookup. Switching to
SipHash closed the door.

>>> **Interview insight**: "Why does Redis use SipHash if it's slower than
>>> Murmur?" Because hashtable performance is dominated by **the slowest case
>>> an attacker can force**, not by the average case. With unkeyed hashing,
>>> a malicious user can pile a billion keys into a single bucket and turn
>>> the server into a tar pit. SipHash makes that attack require breaking the
>>> hash itself.

The seed is generated by `getRandomBytes()` calling `getrandom()` (Linux),
`/dev/urandom`, or as a fallback the address of a stack variable (last
resort). Lives in `dict_hash_function_seed[16]`.

## 5.3 The Load Factor and When We Grow

Two thresholds drive resizing:

```c
// dict.c
#define DICT_FORCE_RESIZE_RATIO 5
static int dict_can_resize = 1;
```

The growth condition is checked in `_dictExpandIfNeeded`:

1. If `used >= size`, **force expand**, regardless of `dict_can_resize`.
2. If `used / size >= DICT_FORCE_RESIZE_RATIO` (= 5), force expand.
3. If `used >= size` AND `dict_can_resize == 1`, expand.

`dict_can_resize` is normally 1, but it's set to 0 during `BGSAVE` and
`BGREWRITEAOF` (when a child process is forked). Reason: resizing means
rehashing, rehashing means modifying memory in the parent, which dirties
COW pages and inflates the child's memory cost. So Redis pauses voluntary
resizing during forked persistence — but not the *forced* resize at 5x
load, which would be a real correctness problem (lookup becoming O(N)).

>>> **Interview insight**: This pause-during-fork dance is one of those
>>> tiny details that turns a textbook hashtable into a production-grade
>>> one. It's the kind of trade-off you see in *every* mature systems
>>> codebase: a clean abstraction with a small carve-out for a real-world
>>> failure mode.

### 5.3.1 The new size

When we grow, we double:

```c
unsigned long dictNextExp(unsigned long size) {
    if (size <= DICT_HT_INITIAL_SIZE) return DICT_HT_INITIAL_EXP;
    // 1, 2, 4, 8, 16, 32, ...
    unsigned long e = DICT_HT_INITIAL_EXP;
    while (1) {
        if ((1UL << e) >= size) return e;
        e++;
    }
}
```

Initial size is 4 buckets. We always double: 4, 8, 16, 32, ..., up to a
maximum of `LONG_MAX / 2`.

## 5.4 The Magic: Incremental Rehashing

Doubling a hashtable means: allocate the new table, walk every entry in the
old table, recompute its bucket in the new table, link it in. For 50 million
keys, that's ~50 ms of pure single-threaded work. **A 50 ms pause is
unacceptable** for Redis.

The trick: **migrate one bucket at a time, on every dict operation**. Plus a
background helper. By the time the table has been touched enough for the
load factor to exceed 1 again, rehashing is already done.

The state machine:

```
  Steady state:                       Rehashing:
  +---------------+                   +-----------------+
  | ht[0]: live   |                   | ht[0]: shrinking|
  | ht[1]: NULL   |                   | ht[1]: growing  |
  | rehashidx: -1 |                   | rehashidx >= 0  |
  +---------------+                   +-----------------+
                                            |
                                            v
                                      All entries migrated;
                                      ht[0] freed; ht[1] -> ht[0]
                                      rehashidx = -1
                                      back to steady state
```

`rehashidx` is the index of the next bucket in `ht[0]` that needs migrating.
When it reaches `ht_size[0]`, rehashing is done.

### 5.4.1 The Per-Operation Migration

`_dictRehashStep()` is called from inside `dictAddRaw`, `dictGenericDelete`,
`dictFind`, `dictGetRandomKey`, etc. — every public mutating or finding
function:

```c
// dict.c
static void _dictRehashStep(dict *d) {
    if (d->pauserehash == 0) dictRehash(d, 1);
}

int dictRehash(dict *d, int n) {
    int empty_visits = n * 10;        // limit how many empty buckets to walk
    if (!dictIsRehashing(d)) return 0;

    while (n-- && d->ht_used[0] != 0) {
        dictEntry *de, *nextde;
        // skip empty buckets
        while (d->ht_table[0][d->rehashidx] == NULL) {
            d->rehashidx++;
            if (--empty_visits == 0) return 1;
        }
        de = d->ht_table[0][d->rehashidx];
        while (de) {
            uint64_t h;
            nextde = de->next;
            // recompute bucket in the *new* table
            h = dictHashKey(d, de->key) & DICTHT_SIZE_MASK(d->ht_size_exp[1]);
            de->next = d->ht_table[1][h];
            d->ht_table[1][h] = de;
            d->ht_used[0]--;
            d->ht_used[1]++;
            de = nextde;
        }
        d->ht_table[0][d->rehashidx] = NULL;
        d->rehashidx++;
    }

    if (d->ht_used[0] == 0) {     // all done
        zfree(d->ht_table[0]);
        d->ht_table[0] = d->ht_table[1];
        d->ht_used[0] = d->ht_used[1];
        d->ht_size_exp[0] = d->ht_size_exp[1];
        d->ht_table[1] = NULL;
        d->ht_used[1] = 0;
        d->ht_size_exp[1] = -1;
        d->rehashidx = -1;
        return 0;
    }
    return 1;     // more to do
}
```

Each call migrates **one non-empty bucket** (which may contain a chain of
entries — usually 1-3, never more than ~10 at the load factor limit). The
cost is O(chain length) ≈ O(1) amortised.

### 5.4.2 The Background Helper

Per-operation rehashing is great when the dict is busy. What if it isn't?
A dict could get stuck mid-rehash for a long time if no one touches it —
keeping ht[1] allocated and slowing every read.

So `serverCron` runs `incrementallyRehash(db)` every tick, which calls
`dictRehashMilliseconds(d, 1)` — rehash for up to 1 millisecond at a time:

```c
// dict.c
int dictRehashMilliseconds(dict *d, int ms) {
    if (d->pauserehash > 0) return 0;
    long long start = timeInMilliseconds();
    int rehashes = 0;
    while (dictRehash(d, 100)) {
        rehashes += 100;
        if (timeInMilliseconds() - start > ms) break;
    }
    return rehashes;
}
```

`serverCron` runs every `1000/server.hz` ms (default 100 ms). Every tick
we burn at most 1 ms on rehashing. Worst case extra latency: 1 ms per 100 ms,
or 1% of CPU.

>>> **Interview insight**: This is the crucial detail. Incremental rehashing
>>> isn't *just* "amortise on operations." It's amortise on operations *and*
>>> bound the cron-driven catchup to 1 ms per tick. Both legs are needed: the
>>> first keeps a busy dict moving; the second prevents an idle dict from
>>> stalling forever in mid-rehash.

### 5.4.3 The Pause Counter

`pauserehash` lets specific operations temporarily prevent rehashing without
fully disabling it. When `dictBlockedKey *db; ...` is iterating, you don't
want buckets moving under your feet. So:

```c
dictPauseRehashing(d);
// ... iterate ht[0] ...
dictResumeRehashing(d);
```

The pause is reentrant (counter, not boolean) so nested operations are safe.
`dictRehash` and `_dictRehashStep` both check `pauserehash == 0` before
doing anything.

## 5.5 What Lookups Look Like During Rehashing

```c
dictEntry *dictFind(dict *d, const void *key) {
    if (dictSize(d) == 0) return NULL;
    if (dictIsRehashing(d)) _dictRehashStep(d);
    uint64_t h = dictHashKey(d, key);
    for (int table = 0; table <= 1; table++) {
        unsigned long idx = h & DICTHT_SIZE_MASK(d->ht_size_exp[table]);
        dictEntry *he = d->ht_table[table][idx];
        while (he) {
            if (key == he->key || dictCompareKeys(d, key, he->key)) return he;
            he = he->next;
        }
        if (!dictIsRehashing(d)) return NULL;
    }
    return NULL;
}
```

Logic:

1. Step the rehash forward by one bucket (amortise).
2. Hash the key once.
3. Look in `ht[0]` first. If found, return.
4. If we're rehashing AND ht[0] miss, look in `ht[1]`.
5. If we find nothing in either, return NULL.

The cost of a lookup during rehashing is *at most* twice the cost outside
rehashing — and in practice less, because once a bucket has been migrated
its `ht[0]` slot is NULL (a single pointer check, no chain walk).

### 5.5.1 What Inserts Look Like During Rehashing

`dictAddRaw` inserts into `ht[1]` if rehashing, else into `ht[0]`. Why?
Because we never want `ht[0]` to grow during rehashing — the whole point of
rehashing is to shrink it to zero. Putting new entries in `ht[1]` means
"the rehash makes monotonic progress."

```c
dictEntry *dictAddRaw(dict *d, void *key, dictEntry **existing) {
    if (dictIsRehashing(d)) _dictRehashStep(d);
    int idx = _dictKeyIndex(d, key, dictHashKey(d, key), existing);
    if (idx == -1) return NULL;
    dictht *ht = dictIsRehashing(d) ? &d->ht[1] : &d->ht[0];
    dictEntry *entry = zmalloc(sizeof(*entry));
    entry->next = ht->table[idx];
    ht->table[idx] = entry;
    ht->used++;
    dictSetKey(d, entry, key);
    return entry;
}
```

### 5.5.2 What Deletes Look Like During Rehashing

Same shape — check `ht[0]` first, then `ht[1]`. Decrement the right
`ht_used`.

## 5.6 The Reverse-Bit SCAN Cursor

Now we get to the cleverest piece of dict code: how `SCAN` walks the
keyspace **safely across rehashes** without missing or duplicating keys.

The naive approach — "iterate buckets 0, 1, 2, ..., ht_size-1" — fails when
the dict resizes mid-scan: buckets get renumbered, your cursor is meaningless,
keys get visited twice or skipped.

Redis solves this with a cursor that visits buckets in **reverse-bit binary
counting order**. The cursor isn't an index — it's a permuted index that
behaves correctly under doubling and halving.

### 5.6.1 The intuition

Suppose we have a 4-bucket table and we scan with cursor = 0, 1, 2, 3 in
order. Now the table doubles to 8 buckets. Bucket `i` in the old table
splits into buckets `i` and `i + 4` in the new table (because the new
size's mask is one bit wider, and the new bit was previously ignored).

If we naively continued counting after bucket 1, we'd visit 2, 3 in the new
table — *missing* the new bucket 5 (which contains entries that were in old
bucket 1 but happen to have the new high bit set).

The reverse-bit trick: visit buckets in the order their bit-reversed indices
are 0, 1, 2, 3 — i.e., 000 → 100 → 010 → 110 → 001 → 101 → 011 → 111.

Why? Because when the table doubles, bucket `i` in the old size becomes
buckets `i` and `i + N/2` in the new size. In reverse-bit ordering, those
two buckets are consecutive — so we visit them next, never skipping a key.

### 5.6.2 The cursor advance

```c
v = cursor;
v |= ~m0;            // OR with the inverted mask of the OLD table size
v = rev(v);          // reverse all bits
v++;                 // increment as if it were normal counting
v = rev(v);          // reverse back
return v;
```

`m0 = size - 1` (the bucket mask). `~m0` is the high bits that aren't part of
the index. `rev` reverses all 64 bits.

The full algorithm in `dictScan`:

```c
unsigned long dictScan(dict *d, unsigned long v,
                       dictScanFunction *fn,
                       dictScanBucketFunction *bucketfn,
                       void *privdata) {
    if (dictSize(d) == 0) return 0;

    dictPauseRehashing(d);

    if (!dictIsRehashing(d)) {
        // simple case: only one table
        m0 = DICTHT_SIZE_MASK(d->ht_size_exp[0]);
        // emit bucket v in ht[0]
        emitBucket(d->ht_table[0][v & m0]);
        // advance cursor
        v |= ~m0;
        v = rev(v); v++; v = rev(v);
    } else {
        // rehashing case: visit both tables
        // ensure t0 is the smaller table
        if (DICTHT_SIZE(d->ht_size_exp[0]) > DICTHT_SIZE(d->ht_size_exp[1])) swap(t0, t1);
        m0 = DICTHT_SIZE_MASK(d->ht_size_exp[t0]);
        m1 = DICTHT_SIZE_MASK(d->ht_size_exp[t1]);

        // emit bucket v in the smaller table
        emitBucket(d->ht_table[t0][v & m0]);
        // ALSO emit all buckets in the larger table that share the v's
        // low bits (i.e., that came from this t0 bucket during rehash)
        do {
            emitBucket(d->ht_table[t1][v & m1]);
            v |= ~m1;
            v = rev(v); v++; v = rev(v);
        } while (v & (m0 ^ m1));

        // advance cursor for next call
        v |= ~m0;
        v = rev(v); v++; v = rev(v);
    }

    dictResumeRehashing(d);
    return v;
}
```

The user calls `SCAN 0 MATCH ...`, gets back `(cursor, [keys])`, then calls
`SCAN <cursor> ...` again until cursor returns 0.

### 5.6.3 What SCAN guarantees

- Every key that was present **for the entire scan** will be returned
  exactly once.
- Keys added during the scan **may or may not** be returned.
- Keys removed during the scan **may or may not** be returned.
- Keys may be returned multiple times **only** if the dict shrunk during
  the scan (rare; the reverse-bit invariant for shrinking is weaker).
- The cursor is **not stateful on the server** — it's just a number.

The application is expected to dedupe (e.g., insert into a `Set` and
deduplicate). The MATCH pattern is applied **after** the bucket walk, so
specifying `MATCH user:*` on a 100M-key DB still walks every bucket — only
the *output* is filtered. Use `COUNT` to control how many buckets per call.

>>> **Interview insight**: "Why does Redis's SCAN not give you a snapshot
>>> view?" Because doing so would either require a global lock (single-thread
>>> killer) or MVCC machinery (Redis doesn't have any). The reverse-bit cursor
>>> is the *minimum* you can get away with: bucket-stable iteration with
>>> small allowed weirdness on entries added/removed during the scan.

## 5.7 The dict in Use: A Tour

Some of the major dicts in a running Redis server:

| Where | dict purpose | Key | Value |
|-------|--------------|-----|-------|
| `server.db[N].dict` | Top-level keyspace per database | SDS key | `robj*` |
| `server.db[N].expires` | Per-key expiration time | SDS key | int64 (ms timestamp) |
| `server.db[N].watched_keys` | Keys WATCHed in MULTI sessions | SDS key | list of clients |
| `server.db[N].blocking_keys` | Keys with blocked clients (BLPOP, etc.) | SDS key | list of clients |
| Each Redis hash with > listpack-max-entries | The hash itself | SDS field | `robj*` (value) or SDS |
| Each Redis set with > intset-max-entries (non-int) | The set itself | SDS member | NULL |
| Each sorted set | Score lookup table | SDS member | `double*` (score) |

**Watch the multiplication.** A Redis with 1 million top-level keys may have
~1.5 million dict instances total because every hash, set, and sorted set is
its own dict.

## 5.8 Real-World Performance Notes

### 5.8.1 Active Defrag and the Dict

Redis 4.0 added "active defragmentation": background scanning of the dict
to relocate entries to less-fragmented memory regions. The scanner uses the
SCAN cursor (yes, the same one) and works exactly because of the reverse-bit
property.

Configuration:
```
activedefrag yes
active-defrag-ignore-bytes 100mb
active-defrag-threshold-lower 10
active-defrag-threshold-upper 100
active-defrag-cycle-min 5
active-defrag-cycle-max 75
```

Active defrag uses jemalloc-specific extensions (`je_get_defrag_hint`) to
detect "this allocation is in a fragmented page." If so, it allocates a
fresh copy and updates the dict's pointer. Costs CPU but recovers RAM.

### 5.8.2 Hashtable resizes during BGSAVE

Recall: `dict_can_resize = 0` during fork. This is to minimise COW dirtying.
But: if a key `INCR`s, that's a write to the value (an integer in the same
allocation as the dict entry), which dirties that page anyway. So the
optimisation is less effective for write-heavy workloads.

Mitigations:
- Keep the working set small enough that fork doesn't need to dirty much.
- Use `hz 100` (default 10) to make `serverCron` more responsive — slightly
  smaller bursts of work.
- If you can tolerate brief pauses, prefer RDB over AOF rewrite (less
  frequent forks).

### 5.8.3 Hash collisions in practice

With SipHash and a load factor < 1, collision chains are typically 1-2
entries long. At load factor 5 (the forced-resize threshold), expected
chain length is ~5. The 99th-percentile chain length under SipHash with
1 million keys is < 12. In other words: even at the worst-case forced
resize state, lookups are ~10 hash compares. Still effectively O(1).

## 5.9 What `dict` Doesn't Do

Things you might expect from a "real" hashtable but `dict` does not provide:

- **Iteration order**. Insertion order is *not* preserved. (For ordered
  iteration use sorted sets.)
- **Concurrent access**. Single-threaded; no internal locking.
- **Bulk operations**. There's no `dictAddAll` or batch insert. Every key
  goes through one call.
- **Persistence-friendliness**. RDB serialises by walking the dict; the
  dict itself is in-memory only.
- **Open addressing / probing**. Always chaining. Slightly more memory per
  entry; simpler resize logic.

These are all deliberate trade-offs that flow from "single-threaded,
in-memory, value-oriented." A different design (e.g., LMDB's B-Tree, or
DragonflyDB's lock-free shard hashing) chooses differently because it has
different constraints.
