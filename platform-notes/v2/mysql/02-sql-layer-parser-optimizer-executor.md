# Chapter 2: The SQL Layer — Parser, Optimizer, Executor

> **Core question**: How does MySQL turn `SELECT *` into a physical access plan,
> and what happens at each stage between the text you type and the rows you receive?

The SQL layer is the brain of MySQL. InnoDB is the muscle — it stores pages, manages
transactions, and handles crash recovery — but it does not understand SQL. The SQL layer
parses your text into a tree, rewrites that tree into a more efficient form, estimates
costs across thousands of possible execution plans, picks one, and executes it by pulling
rows through an iterator pipeline. Every slow query you have ever debugged was, at some
point, a decision made in this layer.

This chapter traces a query through the full pipeline: lexer, parser, resolver, transformer,
optimizer, executor. Every section references the MySQL 8.0 source code.

```
  ┌────────────────────────────────────────────────────────────────────────┐
  │                        SQL LAYER PIPELINE                             │
  │                                                                       │
  │   SQL Text                                                            │
  │     │                                                                 │
  │     ▼                                                                 │
  │   ┌──────────────┐    ┌──────────────┐    ┌──────────────────────┐    │
  │   │   LEXER      │───▶│   PARSER     │───▶│   NAME/TYPE         │    │
  │   │ sql_lex.cc   │    │ sql_yacc.yy  │    │   RESOLUTION        │    │
  │   │ Tokenize     │    │ Bison grammar│    │ setup_fields()      │    │
  │   └──────────────┘    └──────────────┘    │ setup_tables()      │    │
  │                                           └──────────┬───────────┘    │
  │                                                      │                │
  │                                                      ▼                │
  │   ┌──────────────┐    ┌──────────────┐    ┌──────────────────────┐    │
  │   │  EXECUTOR    │◀───│  COST-BASED  │◀───│  QUERY              │    │
  │   │  Iterator    │    │  OPTIMIZER   │    │  TRANSFORMATIONS    │    │
  │   │  pipeline    │    │  join_optimizer│   │  Subquery→semijoin │    │
  │   │  (pull model)│    │  range_opt   │    │  derived merging    │    │
  │   └──────────────┘    └──────────────┘    └──────────────────────┘    │

  │         ▼                                                             │
  │   Result set → network buffer → client                                │
  └────────────────────────────────────────────────────────────────────────┘

## 2.1 Lexical Analysis and Parsing

### 2.1.1 The Lexer — `sql_lex.cc`

The lexer (lexical analyzer) converts raw SQL text into a stream of tokens. MySQL's lexer
lives in `sql/sql_lex.cc` and is hand-written, not generated by Flex. This is a deliberate
choice: MySQL's lexical rules are context-sensitive in ways that Flex handles poorly.

The core function is `MYSQLlex()`, called repeatedly by the Bison parser. Each invocation
returns one token — an integer constant like `SELECT_SYM`, `IDENT`, `NUM`, `TEXT_STRING` —
and stores the token's value in a `YYSTYPE` union.

**Tokenization walk-through for `SELECT id FROM users WHERE age > 30`:**

```
Input:   S E L E C T   i d   F R O M   u s e r s   W H E R E   a g e   >   3 0
         ─────────────┬───── ──┬── ─────┬──── ──────┬───── ──────┬── ──┬── ──┬──
Token:   SELECT_SYM   IDENT  FROM     IDENT   WHERE_SYM  IDENT  GT  NUM
         (keyword)    ("id") (keyword)("users")(keyword)  ("age")     (30)
```

**Keyword handling.** MySQL has ~750 reserved words. The lexer uses a hash-based lookup
in `sql/lex_hash.h` (generated at build time by `gen_lex_hash.cc`) to determine if an
identifier is a keyword. Reserved words cannot be used as unquoted identifiers; non-reserved
words (like `ACTION`, `AFTER`) can.

>>> **Interview insight**: MySQL distinguishes between *reserved* and *non-reserved* keywords.
>>> `SELECT` is reserved — you cannot name a column `SELECT` without backticks. But `STATUS`,
>>> `EVENT`, and `PARTITION` are non-reserved — they work as identifiers without quoting. This
>>> is a common source of migration bugs when upgrading MySQL versions, as words may become
>>> reserved in newer releases.

**Character set handling.** The lexer must handle multi-byte character sets (UTF-8, UTF-8MB4,
Shift-JIS). The function `my_mbcharlen()` tells the lexer how many bytes the current character
occupies. This is checked for every byte of input. For UTF-8MB4, a single character can span
1-4 bytes, and the lexer must not split a multi-byte character across token boundaries.

**String literals.** The lexer handles single-quoted strings, double-quoted strings (when
`ANSI_QUOTES` is off), national character strings (`N'text'`), hex literals (`0x4D7953514C`),
and bit-value literals (`b'101'`). Escape sequences (`\'`, `\\`, `\n`) are processed inline.

**Comments.** Three comment styles are recognized:
- `-- comment` (SQL standard; note the required space after `--`)
- `# comment` (MySQL extension)
- `/* comment */` (C-style; also used for optimizer hints: `/*+ ... */`)

The hint-comment parser (`sql/sql_hints.yy`) is a separate Bison grammar embedded within
C-style comments. When the lexer sees `/*+`, it switches to the hint parser.

### 2.1.2 The Parser — `sql_yacc.yy`

MySQL's parser is a Bison-generated LALR(1) parser. The grammar file `sql/sql_yacc.yy` is
the largest open-source Bison grammar in existence — approximately 17,000 lines of grammar
rules defining the complete MySQL SQL dialect.

**Source files involved:**
- `sql/sql_yacc.yy` — the grammar (input to Bison)
- `sql/sql_yacc.cc` — generated parser (output of Bison, ~250,000 lines of C++)
- `sql/parse_tree_nodes.h`, `sql/parse_tree_items.h` — parse tree node definitions

**Grammar structure (simplified):**

```
query:
    verb_clause ';'
    ;

verb_clause:
    select_stmt
  | insert_stmt
  | update_stmt
  | delete_stmt
  | ...
  ;

select_stmt:
    SELECT select_options select_item_list
    FROM table_reference_list
    opt_where_clause
    opt_group_clause
    opt_having_clause
    opt_order_clause
    opt_limit_clause
    ;
```

Each grammar rule has an action block (C++ code in `{ ... }`) that constructs a parse tree
node. For example, the `select_stmt` rule creates a `PT_select_stmt` object and attaches
child nodes for the item list, table references, and clauses.

### 2.1.3 Parse Tree Construction

The parser builds a tree of `Parse_tree_node` objects. Key node types:

| Node Class              | Represents                           | Source File                  |
|-------------------------|--------------------------------------|------------------------------|
| `PT_select_stmt`        | Complete SELECT statement            | `parse_tree_nodes.h`         |
| `PT_table_reference`    | A single table in FROM clause        | `parse_tree_nodes.h`         |
| `PT_joined_table`       | A JOIN between two table references  | `parse_tree_nodes.h`         |
| `PT_select_item_list`   | The column list after SELECT         | `parse_tree_nodes.h`         |
| `PT_where`              | WHERE clause                         | `parse_tree_nodes.h`         |
| `PT_order_list`         | ORDER BY clause                      | `parse_tree_nodes.h`         |
| `PT_group`              | GROUP BY clause                      | `parse_tree_nodes.h`         |
| `PT_limit_clause`       | LIMIT / OFFSET                       | `parse_tree_nodes.h`         |
| `Item`                  | Any expression (base class)          | `item.h`                     |
| `Item_field`            | A column reference like `t.id`       | `item.h`                     |
| `Item_func`             | A function call or operator          | `item_func.h`                |
| `Item_subselect`        | A subquery expression                | `item_subselect.h`           |

**Parse tree for `SELECT u.name FROM users u WHERE u.age > 30`:**

```
PT_select_stmt
├── PT_select_item_list
│   └── Item_field("u", "name")     ← unresolved at this point
├── PT_table_reference("users", alias="u")
└── PT_where
    └── Item_func_gt
        ├── Item_field("u", "age")   ← unresolved
        └── Item_int(30)
```

At this stage, `Item_field("u", "name")` is just a name string. The parser does not verify
that table `users` exists, that it has a column `name`, or that the alias `u` is valid.
That is the resolver's job.

### 2.1.4 Name Resolution — `setup_fields()` and `setup_tables()`

After parsing, MySQL enters the *preparation* phase (`SELECT_LEX::prepare()`), which
performs name resolution and type checking. This is where unresolved names are bound to
actual database objects.

**Table resolution** (`setup_tables()` in `sql/sql_base.cc`):
1. For each table reference, open the table definition from the data dictionary (or the
   table definition cache).
2. Verify the user has `SELECT` privilege on the table.
3. Resolve aliases — `FROM users u` maps alias `u` to table `users`.
4. For views, expand the view definition inline (or decide to materialize it).

**Column resolution** (`setup_fields()` in `sql/sql_base.cc`):
1. Walk every `Item_field` in the query.
2. For each, search the table list for a matching column name.
3. If the field has a table qualifier (`u.name`), search only that table.
4. If no qualifier (`SELECT name FROM users, orders`), search all tables — ambiguity here
   is an error.
5. Bind the `Item_field` to the actual `Field` object, which carries the column's data type,
   nullability, character set, and storage position.

>>> **Interview insight**: Name resolution order matters. MySQL resolves outer query names
>>> *after* inner query names. In a correlated subquery like
>>> `SELECT * FROM t1 WHERE x IN (SELECT x FROM t2 WHERE t2.y = t1.y)`, the inner `t2.y`
>>> is resolved first. If `t2` has no column `y`, MySQL looks in the outer scope (`t1.y`).
>>> This implicit outer-reference can cause silent correlated subqueries when you intended
>>> an uncorrelated one — a classic performance trap.

### 2.1.5 Type Resolution and Implicit Conversions

After name resolution, MySQL determines the data type of every expression. This includes
resolving implicit type conversions — and this is where performance disasters hide.

**How implicit conversion works:**

When MySQL compares two values of different types, it follows conversion rules defined in
`item_cmpfunc.cc`. The general rule: **strings are converted to numbers, not vice versa.**

```sql
-- Column: varchar_col VARCHAR(50), indexed
SELECT * FROM t WHERE varchar_col = 12345;
```

MySQL converts `varchar_col` to a number for every row. This disables the index because
the conversion is applied to the column, not the constant. The query becomes a full table
scan.

```sql
-- The fix: match the type
SELECT * FROM t WHERE varchar_col = '12345';
```

**Character set conversion rules** (evaluated in `Item_func::agg_arg_charsets()`):
- If both sides have the same character set, no conversion.
- If one side is a binary string, the other is converted to binary.
- If character sets differ, MySQL uses the *coercibility* rules to decide which side converts.
  A column (coercibility 2) loses to a literal (coercibility 4), so MySQL converts the
  literal to the column's charset — this is safe for indexes.
- But if you explicitly cast a literal with `CONVERT()` or use `_utf8mb4'text'`, the
  coercibility changes, and the conversion may fall on the column side.

>>> **Interview insight**: Three implicit conversion traps that kill indexes:
>>> 1. Comparing a `VARCHAR` column to an integer literal (column is converted)
>>> 2. Comparing columns with different character sets where the wrong side converts
>>> 3. Comparing `DATETIME` column with a string — MySQL calls `CAST()` on the column
>>>    if the string is not in a recognized datetime format
>>> Always use `EXPLAIN` and check for "Using where" without "Using index" as a warning sign.

**`ENUM` internals.** An `ENUM` column stores integers internally (1, 2, 3, ...) but
compares as strings by default. Comparing an ENUM column to an integer forces numeric
comparison, which can give unexpected results: `WHERE enum_col = 0` matches rows where
`enum_col` is an invalid value (stored as 0), not the first enum member.

## 2.2 Query Transformations (Logical Optimization)

Before the cost-based optimizer touches the query, MySQL applies rule-based *logical
transformations* that rewrite the query into an equivalent but more efficient form. These
transformations happen in `sql/sql_optimizer.cc` during the `JOIN::optimize()` call.

### 2.2.1 Subquery Strategies

Subqueries are the most complex transformation target. MySQL 8.0 supports several strategies,
each suited to different subquery shapes.

**Strategy 1: Semi-join transformation.**
For `IN` and `EXISTS` subqueries in the WHERE clause, MySQL can transform the subquery into
a semi-join — an internal join type that returns at most one match per outer row.

```sql
-- Original
SELECT * FROM orders WHERE customer_id IN (SELECT id FROM customers WHERE country = 'US');

-- After semi-join transformation (internal representation)
SELECT orders.* FROM orders SEMI JOIN customers
  ON orders.customer_id = customers.id AND customers.country = 'US';
```

Semi-join execution strategies (chosen by the cost-based optimizer):

| Strategy            | When Used                               | How It Works                                      |
|---------------------|-----------------------------------------|---------------------------------------------------|
| **FirstMatch**      | Small inner table, good index on outer  | For each outer row, scan inner; stop at first match|
| **LooseScan**       | Index on inner table covers join key    | Scan inner index, skip duplicates, probe outer     |
| **DuplicateWeedout**| No good index, but few duplicates       | Execute as join, remove duplicates via tmp table   |
| **MaterializeLookup**| Inner is small and uncorrelated        | Materialize inner into hash table, probe per row   |
| **MaterializeScan** | Inner is small, outer is large          | Materialize inner, scan and probe outer            |

```
  Semi-Join Strategy Selection (optimizer chooses by cost):

  IN-subquery ──▶ Is it correlated? ──Yes──▶ FirstMatch or DuplicateWeedout
                        │
                       No
                        │
                        ▼
                  Can inner be materialized? ──No──▶ FirstMatch
                        │
                       Yes
                        │
                        ▼
                  MaterializeLookup vs MaterializeScan (by cost)
                  vs LooseScan (if index available)
```

>>> **Interview insight**: Prior to MySQL 5.6, `IN` subqueries were executed as *dependent
>>> subqueries* — re-executed for every outer row, even when uncorrelated. This was MySQL's
>>> single most notorious optimizer failure. Semi-join transformation (5.6+) and materialization
>>> fixed this. If you see `DEPENDENT SUBQUERY` in EXPLAIN on MySQL 8.0, it means the
>>> optimizer could not apply semi-join — investigate why (often: the subquery is in the
>>> SELECT list or uses HAVING).

**Strategy 2: Materialization.** For subqueries that cannot be semi-joined (e.g., in the
SELECT list, or with `NOT IN`), MySQL materializes the subquery result into an internal
temporary table, then probes it. The temp table is either in-memory (TempTable engine) or
on-disk (InnoDB internal temp table).

**Strategy 3: EXISTS transformation.** MySQL can transform `IN (SELECT ...)` into
`EXISTS (SELECT 1 FROM ... WHERE ...)` when doing so allows use of an index on the inner
query's WHERE clause. This is the `Item_in_subselect::select_transformer()` path.

### 2.2.2 Derived Table Merging vs Materialization

A *derived table* is a subquery in the FROM clause:

```sql
SELECT d.name, d.total
FROM (SELECT name, SUM(amount) AS total FROM orders GROUP BY name) AS d
WHERE d.total > 1000;
```

MySQL has two strategies:

**Merging** (`derived_merge`): Inline the derived table into the outer query. After merging,
the query becomes equivalent to:

```sql
SELECT name, SUM(amount) AS total FROM orders GROUP BY name HAVING SUM(amount) > 1000;
```

Merging avoids materializing the intermediate result. The optimizer flag
`optimizer_switch='derived_merge=on'` controls this (enabled by default in 8.0).

**Materialization**: Execute the derived table query, store results in a temporary table,
then scan that table for the outer query. Used when merging is impossible — specifically
when the derived table uses:
- `DISTINCT`, `GROUP BY`, `HAVING`, `LIMIT`
- Aggregate functions
- `UNION`
- User variables with side effects

>>> **Interview insight**: A common pitfall is wrapping a query in a derived table for
>>> "readability" and unknowingly preventing index use. If the derived table gets materialized,
>>> the outer WHERE clause cannot push down to use the original table's indexes. Always check
>>> EXPLAIN for `<derived2>` — if you see it with a full scan, the derived table was
>>> materialized and your outer predicate is scanning the temp table.

### 2.2.3 Outer Join Simplification

MySQL can convert an outer join to an inner join when the WHERE clause rejects NULLs on
the outer side. This is called *outer join simplification* or *NULL-rejecting predicate
optimization*.

```sql
-- This LEFT JOIN is semantically equivalent to INNER JOIN:
SELECT * FROM orders o LEFT JOIN customers c ON o.cust_id = c.id
WHERE c.country = 'US';
-- c.country = 'US' rejects NULLs (NULL != 'US'), so unmatched rows are filtered anyway.
-- MySQL converts it to INNER JOIN, enabling more join orderings.
```

Why this matters: LEFT JOIN constrains join order — the left table must be processed before
the right table. An INNER JOIN has no such constraint, giving the optimizer more join
orderings to evaluate.

Implementation: `simplify_joins()` in `sql/sql_optimizer.cc`. It walks the join tree and
checks each outer join's ON condition against the WHERE clause. If any WHERE predicate
references a NULL-extended column with a NULL-rejecting function (comparison, IS NOT NULL,
IN, etc.), the outer join is simplified.

### 2.2.4 Constant Folding and Predicate Pushdown

**Constant folding** (`fold_condition()` in `sql/sql_optimizer.cc`):
- `WHERE 1 = 1 AND col > 5` → `WHERE col > 5`
- `WHERE col > 5 AND col > 3` → `WHERE col > 5`
- `WHERE false` → query returns empty (the optimizer can short-circuit)
- Expressions involving only constants are evaluated at optimization time.

**Predicate pushdown**: Moving WHERE conditions closer to the data source. MySQL pushes
predicates:
1. From outer queries into derived tables (when merging is possible).
2. From the WHERE clause into JOIN...ON clauses (for inner joins only).
3. Through UNION branches when possible.
4. To the storage engine via **Index Condition Pushdown** (ICP) — covered in section 2.3.

### 2.2.5 WHERE / HAVING / ON Clause Optimization

MySQL applies several optimizations to these clauses:

**Equality propagation**: If `WHERE a = b AND b = 5`, MySQL infers `a = 5` and can use an
index on column `a` directly.

**Impossible WHERE detection**: `WHERE col > 10 AND col < 5` is detected as impossible. The
query returns zero rows without accessing any table. EXPLAIN shows `Impossible WHERE`.

**Constant table detection**: A table with zero or one row (e.g., queried by primary key with
a constant) is read during optimization, and its column values become constants for the
rest of the query. EXPLAIN shows `const` in the type column.

### 2.2.6 View Merging vs Materialization

Views follow the same logic as derived tables. A view defined as:

```sql
CREATE VIEW active_users AS SELECT * FROM users WHERE status = 'active';
```

When used in a query:
```sql
SELECT name FROM active_users WHERE age > 30;
```

With **merging** (default when possible), the query becomes:
```sql
SELECT name FROM users WHERE status = 'active' AND age > 30;
```

Both predicates can now use a composite index on `(status, age)`. Without merging, the
view would be materialized first with only the `status` filter, and the `age` filter would
be applied to the temp table — no index use.

Views with `ALGORITHM=TEMPTABLE` force materialization. `ALGORITHM=MERGE` requests merging
but falls back to materialization if the view query uses constructs that prevent merging
(same rules as derived tables).

### 2.2.7 Common Table Expression (CTE) Handling

MySQL 8.0 introduced CTEs (`WITH` clause). A non-recursive CTE is treated like a derived
table — it can be merged or materialized using the same rules.

```sql
WITH high_value AS (
    SELECT customer_id, SUM(amount) AS total FROM orders GROUP BY customer_id
)
SELECT c.name, h.total FROM customers c JOIN high_value h ON c.id = h.customer_id;
```

**Key behavior**: If a CTE is referenced multiple times in the query, MySQL materializes it
once and reuses the temporary table for each reference. This is a significant advantage over
repeating the same derived table, which would be re-evaluated each time.

**Recursive CTEs** (`WITH RECURSIVE`) use an iterative execution model:
1. Execute the *anchor* member (the non-recursive SELECT) into a temp table.
2. Execute the *recursive* member, reading from the temp table and appending new rows.
3. Repeat step 2 until no new rows are produced.
4. The `cte_max_recursion_depth` variable (default 1000) prevents infinite loops.

## 2.3 Cost-Based Optimizer

The cost-based optimizer is the most complex component of the SQL layer. Its job: given a
query with N tables, choose the join order, access method for each table, and join algorithm
that minimizes total execution cost.

### 2.3.1 The Cost Model — `server_cost` and `engine_cost` Tables

MySQL 8.0 externalized the cost model into two tables in the `mysql` schema:

**`mysql.server_cost`** — costs for operations in the SQL layer:

| Cost Name                    | Default  | What It Represents                            |
|------------------------------|----------|-----------------------------------------------|
| `row_evaluate_cost`          | 0.1      | Cost of evaluating a WHERE condition per row   |
| `key_compare_cost`           | 0.05     | Cost of comparing keys during sorting          |
| `memory_temptable_create_cost`| 1.0     | Fixed cost to create an in-memory temp table   |
| `memory_temptable_row_cost`  | 0.1      | Per-row cost for in-memory temp table          |
| `disk_temptable_create_cost` | 20.0     | Fixed cost to create an on-disk temp table     |
| `disk_temptable_row_cost`    | 0.5      | Per-row cost for on-disk temp table            |

**`mysql.engine_cost`** — costs for storage engine operations:

| Cost Name              | Default  | What It Represents                              |
|------------------------|----------|-------------------------------------------------|
| `io_block_read_cost`   | 1.0      | Cost of reading one 16 KB page from disk         |
| `memory_block_read_cost`| 0.25    | Cost of reading one 16 KB page from buffer pool  |

The ratio `io_block_read_cost / memory_block_read_cost = 4.0` encodes the assumption that
disk I/O is 4x more expensive than memory access. On modern NVMe SSDs, this ratio should be
lower (perhaps 1.5-2.0). On spinning disks, it should be higher (10-20).

>>> **Interview insight**: The cost model defaults were calibrated for spinning disks in an
>>> era when InnoDB's buffer pool hit rate was the primary performance determinant. On modern
>>> all-SSD deployments with large buffer pools (90%+ hit rate), the defaults significantly
>>> overpenalize disk-based plans. You can tune these:
>>> ```sql
>>> UPDATE mysql.engine_cost SET cost_value = 1.0 WHERE cost_name = 'io_block_read_cost';
>>> UPDATE mysql.engine_cost SET cost_value = 0.5 WHERE cost_name = 'memory_block_read_cost';
>>> FLUSH OPTIMIZER_COSTS;
>>> ```
>>> However, in practice, very few production deployments tune these. The optimizer usually
>>> picks a reasonable plan because relative costs matter more than absolute values.

### 2.3.2 Access Path Enumeration

For each table in the query, the optimizer considers these access methods (in approximate
order of efficiency):

```
  Access Method Spectrum (best → worst):

  system   const    eq_ref    ref     range    index    ALL
    │        │        │        │        │        │       │
    ▼        ▼        ▼        ▼        ▼        ▼       ▼
  0-1 row  PK=const  PK/UQ   Non-UQ  Index    Full    Full
  in table lookup   join     index   range    index   table
                    lookup   lookup  scan     scan    scan
                                     (idx)   (no data)

  Cost increases ──────────────────────────────────────────▶
```

**system**: Table has zero or one row. Cost: 0.
**const**: Primary key or unique index lookup with a constant. The row is read during
optimization and treated as a constant. Cost: 1 page read.
**eq_ref**: Primary key or unique index lookup with a value from another table (join).
At most one row per lookup. Cost: ~1 page read per outer row.
**ref**: Non-unique index lookup. Returns multiple rows for each key value. Cost:
`key_records * (io_block_read_cost or memory_block_read_cost) + key_records * row_evaluate_cost`.
**range**: Index range scan (e.g., `col BETWEEN 5 AND 10`, `col IN (1,2,3)`).
**index**: Full index scan (reads every entry in the index, no data pages). Useful for
covering index queries.
**ALL**: Full table scan. Reads every row in the clustered index.

The optimizer evaluates the cost of each applicable access method and picks the cheapest.
The function `test_quick_select()` in `sql/opt_range.cc` evaluates range scan options.

### 2.3.3 Join Ordering

For a query joining N tables, there are N! possible join orderings. The optimizer must find
the cheapest one.

**Exhaustive search** (for N <= `optimizer_search_depth`, default 62):
Evaluates all permutations. For small N (N <= 7), this is tractable. The implementation
uses dynamic programming with bitmask enumeration (`Optimize_table_order::best_extension_by_limited_search()`
in `sql/sql_planner.cc`).

**Greedy heuristic** (for large N):
When the number of tables exceeds `optimizer_search_depth`, MySQL uses a greedy approach:
pick the cheapest single table to add next, then pick the cheapest second table given the
first, and so on. This is O(N^2) instead of O(N!).

```
  Join Ordering: Exhaustive vs Greedy

  N = 4 tables: A, B, C, D
  Exhaustive evaluates 4! = 24 orderings:
  ABCD, ABDC, ACBD, ACDB, ADBC, ADCB,
  BACD, BADC, BCAD, BCDA, BDAC, BDCA,
  CABD, CADB, CBAD, CBDA, CDAB, CDBA,
  DABC, DACB, DBAC, DBCA, DCAB, DCBA

  N = 15 tables:
  15! = 1,307,674,368,000 orderings ← impossible
  Greedy: ~15*14/2 = 105 evaluations ← tractable
```

**`optimizer_search_depth`** controls when MySQL switches from exhaustive to greedy.
Setting it lower (e.g., 5) speeds up optimization for complex queries but may miss the
optimal plan. Setting it to 0 tells MySQL to auto-decide based on query complexity.

**Pruning**: Even during exhaustive search, the optimizer prunes branches that cannot
possibly beat the best plan found so far. If a partial plan's cost already exceeds the
best complete plan's cost, that branch is abandoned. This is the key optimization that
makes exhaustive search practical for up to ~10-12 tables.

>>> **Interview insight**: If a query joins 20+ tables (common in reporting systems with
>>> star schemas), MySQL's greedy heuristic may not find the optimal join order. Two
>>> solutions: (1) Use `STRAIGHT_JOIN` to force a specific join order you know is optimal.
>>> (2) Break the query into smaller CTEs that each join fewer tables.

### 2.3.4 Index Selection — Cardinality and Histograms

The optimizer's access method choice depends on *cardinality estimation* — estimating how
many rows a given predicate will match.

**Index statistics**: InnoDB maintains per-index cardinality estimates via random sampling
of index pages. The variable `innodb_stats_persistent` (default ON in 8.0) stores these
in the `mysql.innodb_index_stats` table. The variable `innodb_stats_persistent_sample_pages`
(default 20) controls how many pages are sampled during `ANALYZE TABLE`.

**Cardinality estimation formula** (simplified):

```
Estimated rows = total_rows_in_table / cardinality_of_index
```

For a table with 1,000,000 rows and an index with cardinality 50,000 (50,000 distinct
values), each key lookup estimates `1,000,000 / 50,000 = 20` rows.

**The problem with uniform distribution assumption**: This formula assumes values are
uniformly distributed. If 90% of rows have `status = 'active'` and 10% have
`status = 'inactive'`, the cardinality is 2, and each lookup estimates `1,000,000 / 2 =
500,000` rows. This is correct for the average case but wildly wrong for the specific case.
Querying `status = 'inactive'` would estimate 500,000 rows (should be 100,000), potentially
choosing a full table scan when a range scan would be far better.

This is why MySQL 8.0 introduced **histograms** — see section 2.4.

### 2.3.5 Range Optimization — SEL_TREE and SEL_ARG

The range optimizer (`sql/opt_range.cc`) determines whether a range scan on an index is
beneficial. It represents possible ranges using two data structures:

**`SEL_ARG`**: A single key range on one key part. For example, `col1 BETWEEN 5 AND 10`
creates a SEL_ARG with `min_value=5, max_value=10, min_flag=0, max_flag=0` (inclusive
on both sides). SEL_ARGs form a linked list for OR conditions and a tree for AND conditions
across multiple key parts.

**`SEL_TREE`**: Contains one SEL_ARG tree per index. The range optimizer builds a SEL_TREE
by analyzing the WHERE clause, then evaluates the cost of scanning each index's range.

```
  WHERE key1_part1 > 5 AND key1_part1 < 10 AND key1_part2 = 'abc'
        AND key2_part1 IN (1, 2, 3)

  SEL_TREE:
  ├── Index key1:
  │   └── SEL_ARG(key1_part1: [5, 10])
  │       └── SEL_ARG(key1_part2: ['abc', 'abc'])
  └── Index key2:
      └── SEL_ARG(key2_part1: {1, 2, 3})   ← OR of three equality ranges
```

The optimizer estimates the number of rows in each range by probing index dives
(`handler::records_in_range()`). For each range, InnoDB reads the leftmost and rightmost
leaf pages and estimates the row count between them based on the B+Tree structure.

The variable `eq_range_index_dive_limit` (default 200 in 8.0) controls when MySQL stops
doing index dives for `IN (...)` lists and falls back to index statistics. With a list of
500 values, 500 index dives would be expensive; using the average cardinality is cheaper
(though less accurate).

### 2.3.6 Index Merge Optimization

When no single index covers the entire WHERE clause but multiple indexes each cover a
part, MySQL can use **index merge** — scanning multiple indexes and combining the results.

Three index merge strategies:

**Intersection** (`index_merge_intersection`):
```sql
-- WHERE clause ANDs two conditions on different indexes
SELECT * FROM t WHERE key1 = 10 AND key2 = 20;
-- Scans index on key1 (rows matching key1=10), scans index on key2 (rows matching key2=20),
-- intersects the two sets of primary keys, then fetches rows.
```

**Union** (`index_merge_union`):
```sql
-- WHERE clause ORs two conditions on different indexes
SELECT * FROM t WHERE key1 = 10 OR key2 = 20;
-- Scans both indexes, unions the primary key sets, fetches rows.
```

**Sort-Union** (`index_merge_sort_union`):
Like union but the ranges are not equality conditions, so the primary keys are not in order.
MySQL sorts them to remove duplicates efficiently.

>>> **Interview insight**: Index merge often signals a missing composite index. If you see
>>> `type: index_merge` in EXPLAIN, consider whether a single composite index would eliminate
>>> the need to scan two indexes and merge results. A composite index on `(key1, key2)` would
>>> handle `WHERE key1 = 10 AND key2 = 20` in a single range scan.

### 2.3.7 Multi-Range Read (MRR) Optimization

When a range scan returns primary keys in random order (secondary index), fetching rows from
the clustered index causes random I/O. MRR buffers the primary keys, sorts them, and then
does sequential reads from the clustered index.

```
  Without MRR:                        With MRR:
  Secondary index scan:               Secondary index scan:
  key=5 → PK=892 → fetch page 892    key=5 → PK=892 ─┐
  key=6 → PK=23  → fetch page 23     key=6 → PK=23  ─┤
  key=7 → PK=701 → fetch page 701    key=7 → PK=701 ─┤ Buffer & sort PKs
  key=8 → PK=44  → fetch page 44     key=8 → PK=44  ─┘
                                              │
  Pages fetched: 892, 23, 701, 44             ▼ Sort: 23, 44, 701, 892
  (random I/O)                        Fetch pages: 23, 44, 701, 892
                                      (sequential-ish I/O)
```

MRR is controlled by `optimizer_switch='mrr=on,mrr_cost_based=on'`. The `read_rnd_buffer_size`
variable controls the buffer size for collected primary keys.

### 2.3.8 Batched Key Access (BKA) for Joins

BKA extends MRR to joins. In a nested-loop join where the inner table is accessed via an
index, BKA buffers multiple join key values from the outer table, then does a multi-range
read on the inner table's index.

```
  Standard Nested Loop:          Batched Key Access:
  For each outer row:            Buffer N outer rows:
    Look up inner index           outer_row_1.key ─┐
    Fetch inner row               outer_row_2.key ─┤
    Evaluate                      outer_row_3.key ─┤ MRR batch
    Return if match               ...              ─┤
                                  outer_row_N.key ─┘
  N random lookups                    │
                                      ▼
                                  Sort keys, batch read inner table
                                  Match and return
```

BKA is disabled by default (`optimizer_switch='batched_key_access=off'`). Enable it when
joins on secondary indexes cause high random I/O on the inner table.

### 2.3.9 Index Condition Pushdown (ICP)

Without ICP, MySQL reads a row from the storage engine and then evaluates the WHERE clause
in the SQL layer. With ICP, MySQL pushes parts of the WHERE clause down to the storage
engine, which evaluates them *during* the index scan, before reading the full row from the
clustered index.

```
  Query: SELECT * FROM t WHERE key_part1 = 5 AND key_part2 LIKE '%abc%'
  Index: (key_part1, key_part2)

  Without ICP:
  1. InnoDB: Scan index for key_part1 = 5, return all PKs
  2. InnoDB: Fetch each full row from clustered index
  3. SQL layer: Apply key_part2 LIKE '%abc%' filter  ← many wasted row reads

  With ICP:
  1. InnoDB: Scan index for key_part1 = 5
  2. InnoDB: For each index entry, check key_part2 LIKE '%abc%'
  3. InnoDB: Only fetch full row if index condition passes  ← fewer row reads
```

EXPLAIN shows `Using index condition` in the Extra column when ICP is active. ICP works
with `ref`, `eq_ref`, `ref_or_null`, `range`, `ALL` (for partitioned tables), and
`index_merge` access methods.

>>> **Interview insight**: ICP is one of MySQL's most impactful optimizations, especially
>>> for composite indexes where the leading columns are used for the range but trailing
>>> columns have additional filters. The key insight: ICP can evaluate conditions on *any*
>>> column present in the index, even if that column is not used for the range scan itself.
>>> Design wide composite indexes knowing that ICP will use trailing columns as filters.

## 2.4 Histograms (MySQL 8.0)

Histograms solve the uniform-distribution problem in cardinality estimation. Instead of
assuming values are evenly distributed, a histogram captures the actual distribution of
values in a column.

### 2.4.1 Histogram Types

MySQL 8.0 supports two histogram types:

**Singleton histogram**: Stores every distinct value and its cumulative frequency. Used when
the number of distinct values is <= the number of buckets (default: 100).

```
  Column: status (3 distinct values)
  Singleton histogram:
  ┌─────────────┬────────────────────┐
  │ Value       │ Cumulative Freq    │
  ├─────────────┼────────────────────┤
  │ 'active'    │ 0.75               │  ← 75% of rows
  │ 'inactive'  │ 0.90               │  ← 15% of rows (0.90 - 0.75)
  │ 'suspended' │ 1.00               │  ← 10% of rows
  └─────────────┴────────────────────┘
```

**Equi-height histogram**: Divides the value range into buckets, each containing approximately
the same fraction of rows. Used when distinct values > number of buckets.

```
  Column: age (integer, 0-100, non-uniform)
  Equi-height histogram (4 buckets):
  ┌─────────┬──────────┬──────────────────┬──────────┐
  │ Bucket  │ Range    │ Cumulative Freq  │ Distinct │
  ├─────────┼──────────┼──────────────────┼──────────┤
  │ 1       │ [18, 25] │ 0.25             │ 8        │
  │ 2       │ [25, 32] │ 0.50             │ 8        │
  │ 3       │ [32, 45] │ 0.75             │ 14       │
  │ 4       │ [45, 89] │ 1.00             │ 45       │
  └─────────┴──────────┴──────────────────┴──────────┘
  Bucket 4 spans a wider range because fewer rows have ages 45-89.
```

### 2.4.2 Creating and Updating Histograms

```sql
-- Create histogram with default 100 buckets
ANALYZE TABLE customers UPDATE HISTOGRAM ON country, status;

-- Create histogram with specific bucket count
ANALYZE TABLE orders UPDATE HISTOGRAM ON order_date WITH 50 BUCKETS;

-- Drop a histogram
ANALYZE TABLE customers DROP HISTOGRAM ON country;

-- View histogram data
SELECT HISTOGRAM FROM information_schema.COLUMN_STATISTICS
WHERE TABLE_NAME = 'customers' AND COLUMN_NAME = 'country'\G
```

Histogram creation samples the entire table (reads all rows). For a 100M-row table, this
is expensive — plan for off-peak execution. The histogram is stored as JSON in the data
dictionary and loaded into memory when the table is opened.

Histograms are **not automatically updated**. After significant data changes, you must
re-run `ANALYZE TABLE ... UPDATE HISTOGRAM`. This is a critical operational consideration.

### 2.4.3 How the Optimizer Uses Histograms

When estimating selectivity for a predicate like `WHERE country = 'US'`, the optimizer
checks (in order):
1. If an index exists on `country` — use index statistics (cardinality).
2. If no index but a histogram exists — use the histogram.
3. If neither — use a hardcoded heuristic (e.g., equality on a non-indexed column
   estimates 10% selectivity).

For a singleton histogram, `WHERE country = 'US'` reads the exact frequency from the
histogram. For an equi-height histogram, MySQL interpolates within the matching bucket.

**Range predicates**: For `WHERE age > 40`, the optimizer sums the frequencies of all
buckets with lower bounds above 40, plus a fraction of the bucket containing 40.

### 2.4.4 When Histograms Help vs When They Don't

**Histograms help when**:
- Column is used in WHERE but has no index (and adding an index is too expensive).
- Data distribution is highly skewed (e.g., 90% of orders are `status = 'complete'`).
- Column is used for join cardinality estimation.
- `WHERE col1 = @val AND col2 = @val2` — histograms on both columns help estimate
  the combined selectivity (assuming independence).

**Histograms do NOT help when**:
- An index already exists on the column — index statistics are used instead.
- The column is never used in WHERE/JOIN/GROUP BY.
- The data distribution changes frequently but histograms are not refreshed.
- Correlated columns — histograms assume independence. If `city` and `state` are
  correlated, `WHERE city = 'Chicago' AND state = 'IL'` will underestimate selectivity
  (the optimizer multiplies the two independent estimates).

>>> **Interview insight**: MySQL histograms are single-column only. There are no multi-column
>>> histograms (unlike PostgreSQL's extended statistics). This means for correlated columns,
>>> the optimizer's estimates can be off by orders of magnitude. The workaround: create a
>>> composite index, which gives the optimizer exact multi-column cardinality through index
>>> dives.

### 2.4.5 Interaction with Index Statistics

When both a histogram and an index exist on the same column:
- The optimizer uses **index statistics** (via index dives or cardinality) for access method
  costing, not the histogram.
- Histograms are used only for columns that do not have a usable index in the current query.
- If `eq_range_index_dive_limit` causes the optimizer to skip index dives (large IN lists),
  it falls back to index cardinality, not histograms.

This means histograms and indexes serve different purposes: indexes are for columns you
query frequently (both access method and selectivity), histograms are for columns you
filter on occasionally but where adding an index is not justified.

## 2.5 Optimizer Trace

The optimizer trace is the most powerful tool for understanding why MySQL chose a specific
execution plan. It produces a JSON document recording every decision the optimizer made.

### 2.5.1 Enabling the Trace

```sql
SET optimizer_trace = 'enabled=on';
SET optimizer_trace_max_mem_size = 1048576;  -- 1 MB (default is 1 MB)

-- Run your query
SELECT * FROM orders o JOIN customers c ON o.cust_id = c.id WHERE c.country = 'US';

-- Read the trace
SELECT * FROM information_schema.OPTIMIZER_TRACE\G
```

The trace output is a JSON document with three main steps:
1. `join_preparation` — parsing and name resolution
2. `join_optimization` — the optimizer's work
3. `join_execution` — execution (minimal info here)

### 2.5.2 Reading the JSON Trace

The optimization step is where all the interesting decisions happen. Key sections:

**`rows_estimation`**: For each table, shows the estimated rows for each access method.

```json
{
  "table": "`orders`",
  "rows_estimation": [
    {
      "table": "`orders`",
      "table_scan": {
        "rows": 1000000,
        "cost": 101234.5
      }
    },
    {
      "index": "idx_cust_id",
      "ranges": ["5 <= cust_id <= 5"],
      "rows": 47,
      "cost": 56.2
    }
  ]
}
```

**`considered_execution_plans`**: Shows join orderings evaluated and their costs.

```json
{
  "considered_execution_plans": [
    {
      "plan_prefix": [],
      "table": "`customers`",
      "best_access_path": {
        "considered_access_paths": [
          {"access_type": "ref", "index": "PRIMARY", "rows": 1, "cost": 1.2},
          {"access_type": "scan", "rows": 50000, "cost": 5123.4}
        ]
      },
      "cost_for_plan": 1.2,
      "rows_for_plan": 1,
      "rest_of_plan": [
        {
          "table": "`orders`",
          "best_access_path": {
            "considered_access_paths": [
              {"access_type": "ref", "index": "idx_cust_id", "rows": 47, "cost": 56.2}
            ]
          },
          "cost_for_plan": 57.4,
          "chosen": true
        }
      ]
    }
  ]
}
```

### 2.5.3 Using the Trace to Debug Bad Plans

**Scenario**: A query uses a full table scan when you expect an index range scan.

Step-by-step debugging:
1. Enable optimizer trace.
2. Run the query.
3. In the trace, find the `rows_estimation` section for the problem table.
4. Look for the index you expected — is it listed in `considered_access_paths`?
5. If listed, check its `cost` vs the table scan `cost` — the optimizer chose the lower one.
6. If the index cost is higher than the scan cost, the optimizer believes the range covers
   too many rows. Check `rows` — does the estimate match reality?
7. If `rows` is wrong, check index statistics (`SHOW INDEX FROM table`) and consider
   running `ANALYZE TABLE` or creating a histogram.

### 2.5.4 Practical Example — Full Trace Walkthrough

```sql
-- Setup
CREATE TABLE products (
    id INT PRIMARY KEY,
    category_id INT,
    price DECIMAL(10,2),
    name VARCHAR(200),
    INDEX idx_category (category_id),
    INDEX idx_price (price)
);
-- 1M rows, 100 categories, prices $1-$1000

SET optimizer_trace = 'enabled=on';

SELECT p.name, p.price
FROM products p
WHERE p.category_id = 42 AND p.price > 500.00
ORDER BY p.price;
```

**Trace analysis (key decisions):**

1. **Range analysis** — the optimizer considers:
   - `idx_category`: range `[42, 42]`, estimated 10,000 rows, cost ~12,000
   - `idx_price`: range `(500.00, +inf)`, estimated 500,000 rows, cost ~600,000
   - Index merge (intersection): not considered (different result sets)

2. **Winner**: `idx_category` (cost 12,000 << 600,000).

3. **Sort optimization**: Since the query orders by `price` and uses `idx_category`,
   MySQL must do a filesort. The trace shows `filesort_summary` with the sorting method.

4. **Better design**: A composite index `(category_id, price)` would eliminate the filesort
   (the index already orders by price within each category) and enable ICP for the price
   condition. The trace would show the access path using this composite index with no
   filesort.

## 2.6 Executor — The Iterator Model (MySQL 8.0)

### 2.6.1 Old Execution Model (pre-8.0): The `JOIN::exec()` Monolith

Before MySQL 8.0, query execution lived in a single massive function chain rooted in
`JOIN::exec()`. This function combined join logic, sorting, grouping, temporary table
creation, and result sending into a monolithic code path that was extremely difficult to
modify or extend.

The execution model was *push-based*: the outer loop pushed rows through a chain of
handlers. Each handler decided whether to pass the row along or accumulate it. Adding
a new join type (like hash join) required modifying deeply nested control flow.

Key problems with the old model:
- No clean separation between operators (sort, group, join were entangled).
- `Block Nested Loop` (BNL) was the only non-index join algorithm — no hash join.
- EXPLAIN output and actual execution were computed by different code paths, leading to
  inaccuracies.
- Extending the executor (e.g., adding window functions) required modifying many files.

### 2.6.2 The New Iterator Model

MySQL 8.0.18+ replaced the monolithic executor with an **iterator-based execution model**
(sometimes called the *Volcano model*). Each operator is a `RowIterator` subclass with
three methods:

```cpp
class RowIterator {
  virtual bool Init() = 0;       // Initialize/reset the iterator
  virtual int Read() = 0;        // Return the next row (0=row, -1=EOF, 1=error)
  virtual void UnlockRow() {};   // Release row lock (for UPDATE/DELETE)
};
```

The execution plan is a tree of iterators. The root iterator's `Read()` method is called
repeatedly until it returns EOF. Each `Read()` call *pulls* a row from child iterators,
which in turn pull from their children — hence *pull-based* execution.

### 2.6.3 Iterator Types

| Iterator                    | What It Does                                    | Source File              |
|-----------------------------|-------------------------------------------------|--------------------------|
| `TableScanIterator`         | Sequential scan of all rows in a table          | `iterators/basic_row_iterators.h`|
| `IndexScanIterator`         | Scan all entries of an index                    | `iterators/basic_row_iterators.h`|
| `IndexRangeScanIterator`    | Scan an index range (e.g., key > 5)             | `iterators/basic_row_iterators.h`|
| `RefIterator`               | Index lookup for a specific key value           | `iterators/ref_row_iterators.h`|
| `NestedLoopIterator`        | Nested-loop join: for each outer row, scan inner| `iterators/composite_iterators.h`|
| `HashJoinIterator`          | Hash join: build hash table on inner, probe with outer | `iterators/hash_join_iterator.h`|
| `SortingIterator`           | Sort input rows (filesort)                      | `iterators/sorting_iterator.h`|
| `MaterializeIterator`       | Execute child, store result in temp table        | `iterators/composite_iterators.h`|
| `AggregateIterator`         | Compute aggregates (SUM, COUNT, etc.)           | `iterators/composite_iterators.h`|
| `FilterIterator`            | Apply WHERE condition, pass matching rows        | `iterators/composite_iterators.h`|
| `LimitOffsetIterator`       | Apply LIMIT/OFFSET                              | `iterators/composite_iterators.h`|
| `StreamingIterator`         | Pass rows through without materialization        | `iterators/composite_iterators.h`|
| `WindowIterator`            | Compute window functions                         | `iterators/composite_iterators.h`|
| `ZeroRowsIterator`          | Return no rows (impossible WHERE)               | `iterators/basic_row_iterators.h`|

### 2.6.4 Pipeline Execution — The Pull Model

```
  Iterator Tree for:
  SELECT c.name, SUM(o.amount)
  FROM customers c JOIN orders o ON c.id = o.cust_id
  WHERE c.country = 'US'
  GROUP BY c.name
  ORDER BY SUM(o.amount) DESC
  LIMIT 10

  LimitOffsetIterator (LIMIT 10)
    │
    └── SortingIterator (ORDER BY SUM(o.amount) DESC)
          │
          └── AggregateIterator (GROUP BY c.name, SUM(o.amount))
                │
                └── NestedLoopIterator
                      │
                      ├── [outer] FilterIterator (c.country = 'US')
                      │             │
                      │             └── TableScanIterator (customers)
                      │
                      └── [inner] RefIterator (orders.idx_cust_id, key=c.id)

  Execution flow (pull model):
  1. Root (Limit) calls Read() on Sort
  2. Sort calls Read() on Aggregate until EOF, sorts all groups
  3. Aggregate calls Read() on NestedLoop, accumulates groups
  4. NestedLoop calls Read() on Filter for next outer row
  5. Filter calls Read() on TableScan, applies c.country='US'
  6. For each outer row, NestedLoop calls Read() on RefIterator(orders)
  7. RefIterator does index lookup on orders.idx_cust_id = c.id
```

The beauty of this model: each iterator is self-contained. Adding a new operator (like
hash join) only requires implementing a new `RowIterator` subclass — no changes to
existing iterators.

### 2.6.5 EXPLAIN FORMAT=TREE — Reading the Iterator Tree

MySQL 8.0.16+ supports `EXPLAIN FORMAT=TREE`, which shows the actual iterator tree:

```sql
EXPLAIN FORMAT=TREE
SELECT c.name, SUM(o.amount)
FROM customers c JOIN orders o ON c.id = o.cust_id
WHERE c.country = 'US'
GROUP BY c.name
ORDER BY SUM(o.amount) DESC
LIMIT 10\G
```

Output:
```
-> Limit: 10 row(s)
    -> Sort: SUM(o.amount) DESC
        -> Table scan on <temporary>
            -> Aggregate using temporary table
                -> Nested loop inner join
                    -> Filter: (c.country = 'US')  (cost=5123 rows=2500)
                        -> Table scan on c  (cost=5123 rows=50000)
                    -> Index lookup on o using idx_cust_id (cust_id=c.id)
                       (cost=4.2 rows=20)
```

Each indentation level is a child iterator. The leaf iterators (table scans, index lookups)
are at the deepest level. Read bottom-up to understand data flow.

### 2.6.6 EXPLAIN ANALYZE — Actual Execution Statistics

`EXPLAIN ANALYZE` (MySQL 8.0.18+) executes the query and reports actual timing and row
counts alongside the estimates:

```sql
EXPLAIN ANALYZE
SELECT c.name, SUM(o.amount)
FROM customers c JOIN orders o ON c.id = o.cust_id
WHERE c.country = 'US'
GROUP BY c.name
ORDER BY SUM(o.amount) DESC
LIMIT 10\G
```

```
-> Limit: 10 row(s)  (actual time=234.5..234.5 rows=10 loops=1)
    -> Sort: SUM(o.amount) DESC, limit input to 10 row(s) per chunk
       (actual time=234.5..234.5 rows=10 loops=1)
        -> Table scan on <temporary>
           (actual time=230.1..231.2 rows=2487 loops=1)
            -> Aggregate using temporary table
               (actual time=230.1..230.1 rows=2487 loops=1)
                -> Nested loop inner join
                   (cost=52345 rows=50000)
                   (actual time=0.08..198.5 rows=49823 loops=1)
                    -> Filter: (c.country = 'US')
                       (cost=5123 rows=2500)
                       (actual time=0.04..12.3 rows=2487 loops=1)
                        -> Table scan on c
                           (cost=5123 rows=50000)
                           (actual time=0.03..8.5 rows=50000 loops=1)
                    -> Index lookup on o using idx_cust_id (cust_id=c.id)
                       (cost=4.2 rows=20)
                       (actual time=0.03..0.07 rows=20.03 loops=2487)
```

**How to read EXPLAIN ANALYZE output:**
- `actual time=X..Y` — time in milliseconds to return the first row (X) and all rows (Y).
- `rows=N` — actual number of rows returned by this iterator.
- `loops=N` — how many times this iterator was called (for inner side of nested loop).
- Compare `rows` (estimated) with `rows` (actual). Large discrepancies indicate bad
  cardinality estimates — the root cause of most bad plans.

>>> **Interview insight**: `EXPLAIN ANALYZE` is the single most important diagnostic tool
>>> for query performance. The key pattern to look for: an iterator that estimates 100 rows
>>> but actually produces 100,000. That cardinality misestimate cascades upward — the join
>>> above it planned for 100 iterations but ran 100,000, and the sort above that allocated
>>> memory for 100*20 rows but got 100,000*20. Fix the estimate (update statistics, add
>>> histogram, rewrite predicate) and the plan usually fixes itself.

### 2.6.7 Hash Join (MySQL 8.0.18+)

Before 8.0.18, MySQL's only join algorithm for non-indexed joins was **Block Nested Loop**
(BNL): buffer rows from the outer table, scan the inner table, and check each inner row
against all buffered outer rows. This is O(N * M) with a constant factor improvement from
buffering.

MySQL 8.0.18 introduced **hash join**, which builds an in-memory hash table from the smaller
(build) table and probes it with rows from the larger (probe) table. This is O(N + M) —
dramatically faster for large tables without indexes.

```
  Block Nested Loop (old):        Hash Join (8.0.18+):
                                  
  for each block of outer rows:   BUILD phase:
    for each inner row:             for each build_row:
      for each outer row in block:    hash_table[hash(key)] = build_row
        if match: output             
                                  PROBE phase:
  Time: O(N * M / buffer_size)      for each probe_row:
                                      lookup hash_table[hash(key)]
                                      if match: output
                                  
                                  Time: O(N + M)
```

**When hash join is used:**
- The join has no usable index on the inner table.
- The join condition is an equi-join (`ON a.col = b.col`).
- MySQL 8.0.18+ (hash join replaces BNL entirely in 8.0.20+).

**Memory management**: The hash table uses the `join_buffer_size` variable. If the build
table exceeds this, MySQL spills to disk — it partitions both tables by hash value, writes
partitions to disk, and then performs hash join on each partition pair.

**Hash join limitations in MySQL**:
- Only equi-joins (no hash join for range conditions like `a.col > b.col`).
- No bitmap/Bloom filter optimization (unlike PostgreSQL or Oracle).
- Build input is always the smaller table (estimated by the optimizer).

>>> **Interview insight**: Hash join is the single biggest executor improvement in MySQL 8.0.
>>> Before it, queries like `SELECT * FROM large_table a JOIN large_table_b b ON a.col = b.col`
>>> without indexes would take hours via BNL. With hash join, they complete in seconds.
>>> However, MySQL's hash join is basic compared to PostgreSQL's (no parallel hash join,
>>> no right/full outer hash join until 8.0.20+). Know these limitations for comparison
>>> questions.

## 2.7 Prepared Statements

### 2.7.1 Server-Side Prepared Statements

Prepared statements separate query *parsing* from *execution*. The client sends the query
template once, and MySQL parses and partially validates it. Subsequent executions only need
to bind parameters and optimize/execute.

**Wire protocol:**

```
  Client                                  Server
    │                                       │
    │  COM_STMT_PREPARE                     │
    │  "SELECT * FROM users WHERE id = ?"   │
    │──────────────────────────────────────▶│
    │                                       │ Parse → Parse tree
    │  OK: stmt_id=1, param_count=1         │ Store in THD::stmt_map
    │◀──────────────────────────────────────│
    │                                       │
    │  COM_STMT_EXECUTE                     │
    │  stmt_id=1, param[0] = 42            │
    │──────────────────────────────────────▶│
    │                                       │ Bind params
    │                                       │ Optimize (full re-optimize!)
    │                                       │ Execute
    │  Result set                           │
    │◀──────────────────────────────────────│
    │                                       │
    │  COM_STMT_EXECUTE                     │
    │  stmt_id=1, param[0] = 99            │
    │──────────────────────────────────────▶│
    │                                       │ Bind params
    │                                       │ Optimize again (re-optimize!)
    │                                       │ Execute
    │  Result set                           │
    │◀──────────────────────────────────────│
    │                                       │
    │  COM_STMT_CLOSE                       │
    │  stmt_id=1                            │
    │──────────────────────────────────────▶│
    │                                       │ Free parse tree
```

**Key source functions:**
- `Prepared_statement::prepare()` in `sql/sql_prepare.cc` — parses and validates
- `Prepared_statement::execute()` in `sql/sql_prepare.cc` — binds, optimizes, executes
- `mysql_stmt_prepare()` and `mysql_stmt_execute()` — C API entry points

### 2.7.2 Parse Once, Execute Many

The primary benefit of prepared statements is skipping the parse phase on subsequent
executions. For MySQL, parsing is relatively cheap (milliseconds), so the savings are
modest compared to databases with more expensive parsing (Oracle, PostgreSQL with plan
caching).

**What is saved:**
- Lexing and Bison parsing (~0.1-0.5 ms for a simple query).
- Parse tree construction.
- Syntax validation.

**What is NOT saved:**
- Name resolution (re-run each time — tables may have been altered).
- Type checking.
- Optimization (full re-optimization every execution).
- Execution.

### 2.7.3 Parameter Binding and Type Handling

Parameters (`?` placeholders) are sent as typed binary values in `COM_STMT_EXECUTE`. The
client specifies the type of each parameter (integer, string, double, etc.) in the protocol
packet. MySQL creates `Item_param` objects in the parse tree, and each execution replaces
their values.

**Binary protocol advantage**: Parameters are sent in binary format (e.g., an integer as
4 bytes) instead of text format (e.g., `"12345"` as 5 bytes). This saves conversion
overhead and bandwidth. Result sets are also returned in binary format, avoiding the
server-side `sprintf()` and client-side `atoi()` conversions.

**SQL injection prevention**: Because parameters are sent separately from the query text,
there is no possibility of SQL injection — the parameter value is never interpreted as SQL.
This is a strict protocol-level guarantee, not a text-escaping heuristic.

### 2.7.4 Plan Caching — Or Lack Thereof

This is MySQL's most significant prepared statement limitation compared to PostgreSQL
and Oracle.

**MySQL**: Re-optimizes every execution. `COM_STMT_EXECUTE` always calls `JOIN::optimize()`.
The optimizer sees the actual parameter values and can use them for range estimation and
index selection.

**PostgreSQL**: Caches the plan after a few executions (generic plan). If the cached plan
is good enough for all parameter values, PostgreSQL skips optimization. If not (detected
via plan cost comparison), it falls back to custom plans.

**Trade-off**:
- MySQL always gets the *optimal plan for the specific parameter values* but pays the
  optimization cost (~0.5-5 ms) every time.
- PostgreSQL sometimes gets a *suboptimal generic plan* but saves the optimization cost.

For OLTP queries with simple plans (point lookups, single-table scans), MySQL's approach
is slightly worse (unnecessary re-optimization). For analytical queries where different
parameter values should use different plans (e.g., a selective vs non-selective range),
MySQL's approach is better.

>>> **Interview insight**: "Does MySQL cache prepared statement plans?" is a common interview
>>> question. The answer: **No**. MySQL re-optimizes every execution. This is actually an
>>> advantage for data-skew scenarios — `WHERE status = 'active'` (90% of rows, should scan)
>>> vs `WHERE status = 'suspended'` (0.1%, should use index) will get different plans. In
>>> PostgreSQL, a cached generic plan would use the same plan for both, potentially choosing
>>> poorly for one value. The downside is higher CPU usage from repeated optimization.

### 2.7.5 `useServerPrepStmts=true` in JDBC

The MySQL JDBC connector (Connector/J) has two prepared statement modes:

**Client-side prepared statements** (default, `useServerPrepStmts=false`):
- The connector interpolates parameter values into the SQL text client-side.
- Sends the full SQL text via `COM_QUERY` (text protocol).
- No server-side parse tree is reused.
- Parameters are escaped as text strings.

**Server-side prepared statements** (`useServerPrepStmts=true`):
- Uses `COM_STMT_PREPARE` / `COM_STMT_EXECUTE` protocol.
- Parse tree reused across executions.
- Binary protocol for parameters and results.
- Saves ~0.1-0.5 ms per execution (parsing) + bandwidth savings from binary protocol.

**When to enable server-side prepared statements:**
- High-throughput OLTP with simple, repeated queries (point lookups, inserts).
- Applications executing the same query thousands of times per second.
- When binary protocol bandwidth savings matter (large result sets with many numeric columns).

**When NOT to:**
- One-off analytical queries (no reuse, extra round trip for PREPARE is wasted).
- Applications using a large number of distinct query templates (each PREPARE allocates
  server memory; controlled by `max_prepared_stmt_count`, default 16,382).

**Connection pool interaction**: With HikariCP or similar pools, prepared statements are
cached per connection. When a connection is returned to the pool and reused by a different
thread, the prepared statements from the previous thread remain valid on the server side
if the connection is the same MySQL session.

```
  HikariCP + Server-Side Prepared Statements:

  Thread A                Pool               MySQL Server (Connection 5)
    │                       │                         │
    │  getConnection()      │                         │
    │──────────────────────▶│  (returns conn 5)       │
    │                       │                         │
    │  prepareStatement()   │                         │
    │  (SQL hash → cache hit or COM_STMT_PREPARE)     │
    │──────────────────────────────────────────────────▶
    │                       │                   Parse, store stmt_id=1
    │  executeQuery(params) │                         │
    │──────────────────────────────────────────────────▶
    │                       │                   Bind, optimize, execute
    │  close() → return     │                         │
    │──────────────────────▶│                         │
    │                       │   stmt_id=1 remains valid
    │                       │                         │
  Thread B                  │                         │
    │  getConnection()      │                         │
    │──────────────────────▶│  (returns conn 5)       │
    │  prepareStatement()   │                         │
    │  (same SQL → cache hit, reuse stmt_id=1)        │
    │  executeQuery(params) │                         │
    │──────────────────────────────────────────────────▶
    │                       │                   Bind, optimize, execute
```

## 2.8 Query Cache — Post-Mortem

The query cache was MySQL's attempt at a results cache. It was introduced in MySQL 4.0 and
removed entirely in MySQL 8.0. Understanding why it was removed is as instructive as
understanding the systems that replaced it.

### 2.8.1 How It Worked

The query cache stored `(query_text, result_set)` pairs. When a SELECT arrived, MySQL
hashed the query text and looked it up in the cache. On a hit, the stored result set was
returned immediately — no parsing, optimization, or execution.

```
  Query Cache Architecture (MySQL 5.7 and earlier):

  COM_QUERY("SELECT name FROM users WHERE id = 5")
       │
       ▼
  ┌─────────────────────────────────────┐
  │  Hash(query_text) → cache lookup    │
  │                                     │
  │  Hit? ──Yes──▶ Return cached result │
  │    │                                │
  │   No                                │
  │    │                                │
  │    ▼                                │
  │  Parse → Optimize → Execute         │
  │    │                                │
  │    ▼                                │
  │  Store result in cache              │
  │  Key: exact query text              │
  │  Value: serialized result set       │
  └─────────────────────────────────────┘
```

**Strictness of matching**: The cache key was the *exact byte string* of the query. These
were all cache misses relative to each other:
- `SELECT name FROM users WHERE id = 5`
- `select name from users where id = 5`
- `SELECT  name FROM users WHERE id = 5` (extra space)
- `SELECT name FROM users WHERE id = 5 -- comment`

No normalization of SQL text was performed.

### 2.8.2 Why It Was Removed

The query cache suffered from three fatal design problems:

**Problem 1: Global mutex (`LOCK_query_cache`)**

Every cache lookup and every cache modification required acquiring `LOCK_query_cache`, a
single global mutex. On multi-core systems with high concurrency, this mutex became a
severe bottleneck. Even read-only workloads serialized on this lock because every SELECT
checked the cache.

Symptoms: `Waiting for query cache lock` appeared in `SHOW PROCESSLIST` under high load.
Disabling the query cache (`query_cache_type = OFF`) immediately improved throughput on
systems with many cores.

**Problem 2: Table-level invalidation**

Any write to a table (`INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`, `ALTER`) invalidated *all*
cached results involving that table. Not just the affected rows — all queries touching that
table.

```
  Cache entries:
  Q1: SELECT * FROM users WHERE id = 1      ← cached
  Q2: SELECT * FROM users WHERE id = 99999  ← cached
  Q3: SELECT * FROM orders WHERE total > 100 ← cached

  INSERT INTO users (id, name) VALUES (50000, 'New User');

  Invalidated: Q1, Q2 (both touch 'users')
  Kept: Q3 (does not touch 'users')
```

For write-heavy workloads, the cache was constantly being populated and immediately
invalidated — pure overhead with no benefit.

**Problem 3: Memory fragmentation**

The query cache used its own memory allocator (`query_cache_mem`), which suffered from
fragmentation over time. As results of varying sizes were cached and evicted, the free
memory became fragmented into small unusable blocks. The `Qcache_free_blocks` status
variable would grow while `Qcache_free_memory` remained high — memory was "free" but
unusable.

**The removal**: MySQL 8.0.0 (development milestone) deprecated the query cache.
MySQL 8.0.3 removed it entirely. The `query_cache_type` and `query_cache_size` variables
no longer exist.

### 2.8.3 What Replaced It

The query cache's functionality was not replaced by a single feature. Instead, the caching
problem was decomposed into layers, each handled by the most appropriate component:

**1. ProxySQL query cache:**
ProxySQL (a MySQL-aware proxy) provides a query cache that operates at the proxy layer.
Advantages over MySQL's built-in cache:
- Per-query TTL (time-to-live) instead of table-level invalidation.
- Regex-based query matching (not exact-string).
- No mutex contention inside MySQL.
- Cache shared across backend servers.

```sql
-- ProxySQL: cache SELECT queries matching a pattern for 60 seconds
INSERT INTO mysql_query_rules (rule_id, match_pattern, cache_ttl)
VALUES (1, '^SELECT .* FROM products WHERE category', 60000);
```

**2. Application-level caching (Redis, Memcached):**
For hot-path queries, application-level caching provides:
- Granular invalidation (invalidate specific keys, not entire tables).
- Distributed cache (shared across application instances).
- Flexible TTL policies.
- The application controls when to cache, when to invalidate, and what to cache.

**3. InnoDB buffer pool:**
For I/O-bound workloads, the query cache's real benefit was avoiding disk reads. A properly
sized InnoDB buffer pool achieves the same effect by keeping hot pages in memory. The query
still goes through parse-optimize-execute, but data access is served from RAM.

**4. Read replicas:**
For read-heavy workloads where the query cache masked an overloaded primary, read replicas
distribute the load across multiple servers.

>>> **Interview insight**: "Why was the query cache removed?" is a popular interview question.
>>> The concise answer: it traded a global mutex contention problem for a hit-rate problem.
>>> Under any write load, cache entries were constantly invalidated, so hit rates were low.
>>> Under read-only load, the global mutex serialized all queries. The only workload where
>>> it helped was read-only access to static tables — and those queries are fast anyway because
>>> the data lives in the buffer pool. The lesson: caching must be designed with concurrent
>>> invalidation as a first-class concern, not bolted on with a global lock.

## 2.9 Tying It All Together — A Query's Complete Journey

To solidify the chapter, here is the complete lifecycle of a single query through the SQL layer:

```sql
SELECT c.name, COUNT(o.id) AS order_count
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE c.country = 'US' AND o.created_at > '2024-01-01'
GROUP BY c.name
HAVING COUNT(o.id) > 5
ORDER BY order_count DESC
LIMIT 10;
```

```
  PHASE 1: LEXING
  ┌────────────────────────────────────────────────────────────────┐
  │ MYSQLlex() tokenizes into:                                    │
  │ SELECT_SYM, IDENT("c"), DOT, IDENT("name"), COMMA,           │
  │ COUNT_SYM, LPAREN, IDENT("o"), DOT, IDENT("id"), RPAREN,     │
  │ AS, IDENT("order_count"), FROM, IDENT("customers"),           │
  │ IDENT("c"), JOIN_SYM, IDENT("orders"), IDENT("o"),            │
  │ ON, ... (continues)                                           │
  └────────────────────────────────────────────────────────────────┘
                            │
                            ▼
  PHASE 2: PARSING (Bison)
  ┌────────────────────────────────────────────────────────────────┐
  │ Bison reduces tokens into a parse tree:                       │
  │ PT_select_stmt                                                │
  │ ├── select_item_list: [Item_field(c.name), Item_func_count]   │
  │ ├── table_refs: PT_joined_table(customers c, orders o,        │
  │ │                               ON c.id = o.customer_id)      │
  │ ├── where: Item_cond_and(c.country='US', o.created_at>'...')  │
  │ ├── group_by: [Item_field(c.name)]                            │
  │ ├── having: Item_func_gt(count, 5)                            │
  │ ├── order_by: [Item_field(order_count) DESC]                  │
  │ └── limit: 10                                                 │
  └────────────────────────────────────────────────────────────────┘
                            │
                            ▼
  PHASE 3: NAME & TYPE RESOLUTION
  ┌────────────────────────────────────────────────────────────────┐
  │ setup_tables(): Open 'customers' and 'orders' table defs      │
  │ setup_fields(): Bind c.name → customers.name (VARCHAR(100))   │
  │                 Bind o.id → orders.id (INT)                   │
  │                 Bind c.id → customers.id (INT)                │
  │                 Bind o.customer_id → orders.customer_id (INT) │
  │                 Bind c.country → customers.country (VARCHAR)  │
  │                 Bind o.created_at → orders.created_at (DATE)  │
  │ Type check: '2024-01-01' → implicit CAST to DATE ✓           │
  │ Type check: COUNT(o.id) > 5 → INT > INT ✓                    │
  └────────────────────────────────────────────────────────────────┘
                            │
                            ▼
  PHASE 4: LOGICAL TRANSFORMATIONS
  ┌────────────────────────────────────────────────────────────────┐
  │ • No subqueries to transform                                  │
  │ • INNER JOIN allows both join orderings: (c,o) and (o,c)      │
  │ • Equality propagation: none applicable                       │
  │ • Constant folding: '2024-01-01' is a constant → folded      │
  └────────────────────────────────────────────────────────────────┘
                            │
                            ▼
  PHASE 5: COST-BASED OPTIMIZATION
  ┌────────────────────────────────────────────────────────────────┐
  │ For customers c:                                              │
  │   • ALL (full scan): 50,000 rows, cost = 5123                 │
  │   • ref idx_country('US'): est 2,500 rows, cost = 3050       │
  │   Best: idx_country (ref)                                     │
  │                                                               │
  │ For orders o:                                                 │
  │   • Given c.id: ref idx_customer_id: est 20 rows/key          │
  │   • Range idx_created_at > '2024-01-01': est 300,000 rows     │
  │   Best: idx_customer_id (ref)                                 │
  │                                                               │
  │ Join orderings:                                               │
  │   (c → o): cost = 3050 + 2500*20*4.2 = 213,050               │
  │   (o → c): cost = 300000*5.1 = 1,530,000                     │
  │   Best: c → o                                                 │
  │                                                               │
  │ GROUP BY c.name → needs temp table + sort                     │
  │ ORDER BY order_count DESC → filesort on aggregated result     │
  │ LIMIT 10 → applied after sort                                 │
  └────────────────────────────────────────────────────────────────┘
                            │
                            ▼
  PHASE 6: ITERATOR TREE CONSTRUCTION
  ┌────────────────────────────────────────────────────────────────┐
  │ LimitOffsetIterator (10)                                      │
  │  └── SortingIterator (order_count DESC)                       │
  │       └── FilterIterator (COUNT(o.id) > 5)  ← HAVING         │
  │            └── AggregateIterator (GROUP BY c.name)            │
  │                 └── NestedLoopIterator                        │
  │                      ├── RefIterator(customers, idx_country,  │
  │                      │               key='US')                │
  │                      └── FilterIterator(created_at>'2024-..') │
  │                           └── RefIterator(orders,             │
  │                                idx_customer_id, key=c.id)     │
  └────────────────────────────────────────────────────────────────┘
                            │
                            ▼
  PHASE 7: EXECUTION (pull model)
  ┌────────────────────────────────────────────────────────────────┐
  │ Root.Read() pulls row by row:                                 │
  │ → RefIterator scans idx_country for 'US' customers            │
  │ → For each customer, RefIterator probes orders.idx_customer_id│
  │ → FilterIterator discards orders before 2024-01-01            │
  │ → AggregateIterator groups by c.name, counts o.id             │
  │ → FilterIterator applies HAVING (count > 5)                   │
  │ → SortingIterator collects all groups, sorts by count DESC    │
  │ → LimitOffsetIterator returns first 10 groups                 │
  │ → Result sent to client via network buffer                    │
  └────────────────────────────────────────────────────────────────┘
```

## Summary

The SQL layer transforms text into rows through six distinct phases. The lexer and parser
are mechanically complex but functionally straightforward — they produce a parse tree from
valid SQL. Name resolution binds that tree to real database objects. Logical transformations
rewrite the tree into more efficient forms — subquery flattening, derived table merging,
outer join simplification. The cost-based optimizer evaluates access methods and join orders
using a model calibrated with configurable cost constants and cardinality estimates. The
iterator-based executor (MySQL 8.0) pulls rows through a tree of composable operators.

The three most impactful improvements in MySQL 8.0's SQL layer are: semi-join transformation
(fixing the decades-old subquery performance problem), hash join (providing O(N+M) joins
without indexes), and the iterator execution model (enabling clean extension of the executor).
Understanding these internals is what separates query tuning by intuition from query tuning
by engineering.
