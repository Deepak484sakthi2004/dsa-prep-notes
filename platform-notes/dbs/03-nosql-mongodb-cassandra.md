## 1. WiredTiger Storage Engine

### 1.1 Architecture

WiredTiger (default since MongoDB 3.2) replaced MMAPv1. Key design choices:

```
+-----------------------------------------------------------+
|                    WiredTiger Storage                     |
|                                                           |
|  +-------------------+   +---------------------------+  |
|  | Block Cache        |   | Transaction Manager        |  |
|  | (50% RAM default)  |   | (MVCC via snapshots)       |  |
|  +-------------------+   +---------------------------+  |
|                                                           |
|  +-------------------+   +---------------------------+  |
|  | B+Tree (indexes)  |   | Row Store (document data)  |  |
|  | prefix-compressed |   | variable-length pages      |  |
|  +-------------------+   +---------------------------+  |
|                                                           |
|  +---------------------------------------------------+  |
|  |         On-disk: .wt files per collection          |  |
|  |         Compression: Snappy (default) / zstd       |  |
|  +---------------------------------------------------+  |
+-----------------------------------------------------------+
```

### 1.2 B+Tree for Indexes

WiredTiger uses **B+Trees** for all indexes (unlike Cassandra's LSM). Internal nodes store keys + child pointers; leaf nodes store keys + values (or RIDs for secondary indexes).

**Prefix compression**: for string keys with shared prefixes, only the differing suffix is stored on internal pages → significant space saving for compound string indexes.

**Block cache**: WiredTiger manages its own page cache (`wiredTigerCacheSizeGB`). This is separate from the OS page cache. Total RAM usage ≈ WiredTiger cache + OS file cache + per-connection overhead.

```yaml
# mongod.conf
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 4         # default: 50% of (RAM - 1GB), min 256MB
      journalCompressor: snappy
    collectionConfig:
      blockCompressor: snappy   # zstd for better compression ratio
    indexConfig:
      prefixCompression: true
```

### 1.3 Compression

| Codec | Compression ratio | CPU cost | Use case |
|-------|------------------|----------|---------|
| Snappy | ~2-3x | Low | Default; OLTP |
| zstd | ~4-6x | Medium | Storage-constrained; analytics |
| zlib | ~3-4x | High | Legacy |
| none | 1x | None | Maximum I/O throughput |

## 2. BSON Format and ObjectId

### 2.1 BSON (Binary JSON)

BSON is a binary-encoded serialization of JSON-like documents. Key differences from JSON:
- Length-prefixed strings and documents → O(1) document/field skip
- Rich type system: Date (8B int), int32, int64, Decimal128, Binary, ObjectId
- Arrays stored as embedded documents with string keys ("0", "1", ...)

**BSON overhead**: every key name is stored with every document (unlike relational columns). This is a real storage cost — short field names matter in large collections.

### 2.2 ObjectId Structure (12 bytes)

```
+-------------------+----------+----------+
| timestamp (4B)    | machine  | proc (2B)|   counter (3B)
| unix epoch secs   | id (3B)  |          |
+-------------------+----------+----------+
      ^                    ^                    ^
   auto-extracted       uniqueness          monotonic
   by $natural sort      across hosts       within sec
```

**Key implications**:
- ObjectIds are **roughly monotonic** (sorted by insertion time) → sequential access pattern → B+Tree friendly
- Timestamp embedded → can extract insertion time: `ObjectId.getTimestamp()`
- Generated by client driver (not server) → no round-trip for ID generation

## 3. Indexes

### 3.1 Index Types

| Type | Description | Use case |
|------|-------------|---------|
| Single field | `{field: 1}` | Simple equality/range |
| Compound | `{a: 1, b: -1}` | Multi-field queries; prefix rule |
| Multikey | Index on array field | Query inside arrays |
| Text | Full-text search index | `$text` operator |
| Geospatial (2dsphere) | Geo coordinates | `$near`, `$geoWithin` |
| Wildcard | `{"$**": 1}` | Arbitrary field queries |
| Hashed | `{field: "hashed"}` | Shard key hashing |

### 3.2 Compound Index Prefix Rule

Index `{a: 1, b: 1, c: 1}` can satisfy queries on:
- `{a: 1}`
- `{a: 1, b: 1}`
- `{a: 1, b: 1, c: 1}`

But NOT `{b: 1}` or `{b: 1, c: 1}` — must use leftmost prefix.

**ESR rule for compound index ordering**: Equality fields first, then Sort fields, then Range fields.
```
{status: 1, createdAt: 1, amount: 1}  ← for queries like:
WHERE status = 'active' ORDER BY createdAt RANGE amount > 100
```

### 3.3 Multikey Indexes

When indexing an array field, MongoDB creates one index entry per array element.

```javascript
// Document: {tags: ["java", "spring", "mongodb"]}
// Index on 'tags' → entries: ["java", pk], ["spring", pk], ["mongodb", pk]
```

**Restriction**: only one field in a compound index can be an array field (to prevent index explosion).

### 3.4 Wildcard Indexes

```javascript
db.collection.createIndex({"$**": 1})  // index every field
db.collection.createIndex({"metadata.$**": 1})  // index all sub-fields of 'metadata'
```

Useful for workloads with unpredictable query patterns (e.g., user-defined attributes). WiredTiger stores a separate index entry for every leaf key-value pair in the document.

## 4. Aggregation Pipeline Internals

### 4.1 Pipeline Stages

```
db.orders.aggregate([
  {$match: {status: "active"}},        // filter (uses index if on status)
  {$lookup: {from: "products", ...}},  // join
  {$group: {_id: "$category", sum: {$sum: "$amount"}}},
  {$sort: {sum: -1}},
  {$limit: 10}
])
```

### 4.2 $lookup (Join)

Two implementations:
1. **Hash join** (default since MongoDB 5.0 with index-free foreign collections): builds hash table on the `from` collection, probes with each pipeline document
2. **Index nested loop join**: if `from` collection has an index on the join field → for each input doc, do an index lookup on `from`

MongoDB 5.0+ supports `$lookup` with `let` variables and pipeline for complex joins.

### 4.3 $group (Hash Aggregation)

Builds an in-memory hash table keyed by `_id` expression. If the group result fits in memory (`allowDiskUse: false`, 100MB limit by default) → in-memory. Otherwise → spills to disk.

### 4.4 Index Usage in Aggregation

`$match` uses index if it's the first stage and the filter can use an existing index. `$sort` uses index if sort fields match index and `$match` precedes `$sort`.

```javascript
// Efficient: $match first (index on status), then $sort (if index on {status,createdAt})
[{$match: {status: "active"}}, {$sort: {createdAt: -1}}]

// Inefficient: $sort before $match forces sort on full collection
[{$sort: {createdAt: -1}}, {$match: {status: "active"}}]
```

Use `explain("executionStats")` to verify:
```javascript
db.orders.explain("executionStats").aggregate([...])
```

## 5. Replication — Replica Sets

### 5.1 Architecture

```
+----------+     oplog tailing    +----------+
| PRIMARY  |  ──────────────────> | SECONDARY|
|          |  <── heartbeat ────  |          |
+----------+                      +----------+
     |
     +── heartbeat ──> +----------+
                       | SECONDARY|
                       +----------+
```

- Typically 3+ nodes (odd number for majority calculation)
- **Primary**: accepts all writes
- **Secondary**: replicates from primary via oplog; can serve reads (`readPreference: secondary`)
- **Arbiter**: votes in elections but holds no data; use only when cost of 3rd data node is prohibitive

### 5.2 Oplog — The Replication Log

The **oplog** (`local.oplog.rs`) is a **capped collection** (fixed-size, circular) that records all write operations.

Oplog entry format:
```javascript
{
  ts: Timestamp(1673000000, 1),  // timestamp + ordinal
  h: NumberLong("..."),          // hash
  v: 2,                          // oplog version
  op: "i",                       // "i"=insert, "u"=update, "d"=delete, "c"=command
  ns: "mydb.orders",             // namespace
  o: {_id: ObjectId(...), ...},  // operation payload
  o2: {_id: ObjectId(...)}       // (for updates) filter document
}
```

**Oplog window**: time range covered by the capped oplog. If a secondary is down longer than the oplog window, it cannot resync via oplog → must do a full initial sync. Size: `--oplogSize` (default 5% of disk, max 50GB). Monitor with `rs.printReplicationInfo()`.

### 5.3 Primary Election — Raft-Like Protocol

MongoDB uses a protocol called **"Replication Protocol Version 1" (PV1)** built on top of Raft:

**Election trigger**: primary sends heartbeats every 2 seconds. If a secondary doesn't receive a heartbeat for `electionTimeoutMillis` (default 10 seconds), it calls for an election.

**Steps**:
1. Candidate increments its term (`term++`)
2. Votes for itself; sends `RequestVote` to all other nodes
3. A node grants vote if: (a) it hasn't voted in this term, and (b) candidate's oplog is at least as up-to-date as voter's
4. First to get majority of votes becomes primary
5. New primary sends heartbeats to suppress other candidates

**Safety**: two primaries cannot exist in the same term (guaranteed by vote counting).

**Priority**: `rs.conf()` allows setting priority per node; higher priority nodes prefer to be primary.

### 5.4 Secondary Reads

```javascript
// Client read preference
db.getMongo().setReadPref("secondaryPreferred")
// OR in connection string:
mongodb://host:27017/db?readPreference=secondaryPreferred
```

| Read Preference | Routes to |
|----------------|-----------|
| `primary` | Always primary (default) |
| `primaryPreferred` | Primary if available, else secondary |
| `secondary` | Always secondary (may read stale data) |
| `secondaryPreferred` | Secondary if available, else primary |
| `nearest` | Lowest network latency (may be primary or secondary) |

**Eventual consistency caveat**: secondary reads may be behind by replication lag. Do not use for inventory checks, financial balances — only for analytics, search, reporting.

## 6. Sharding

### 6.1 Architecture

```
Client
  |
mongos (router)  ← reads config from CSRS
  |         \
Shard 1    Shard 2   Shard 3
(replica   (replica  (replica
  set)        set)     set)

CSRS = Config Server Replica Set (stores chunk metadata)
```

### 6.2 Shard Key Selection — The Most Critical Decision

A poor shard key is essentially irreversible without downtime.

**Requirements for a good shard key**:
1. **High cardinality**: many possible values → data can be distributed across many chunks
2. **Even write distribution**: avoid monotonic keys (e.g., ObjectId, auto-increment timestamp) → all writes go to the latest chunk (hotspot), creating a "jumbo chunk" that can't split
3. **Query isolation**: include fields used in most queries so mongos can target specific shards (targeted query vs scatter-gather)

| Shard key | Problem |
|-----------|---------|
| `{_id: 1}` (ObjectId) | Monotonic → all inserts to last chunk (hotspot) |
| `{timestamp: 1}` | Same hotspot problem |
| `{userId: 1}` | Good cardinality; scatter-gather if userId not in query |
| `{userId: 1, _id: 1}` | Compound; good for per-user queries + avoids monotonic hotspot |
| Hashed `{_id: "hashed"}` | Even distribution; all queries are scatter-gather |

**Hashed sharding**: `{field: "hashed"}` — distributes uniformly at cost of making all range queries scatter-gather.

### 6.3 Chunk Migration

- Default chunk size: 128 MB
- Balancer runs in background to equalize chunk counts across shards
- A chunk is split when it grows beyond the max size
- Migration: chunk data is copied from source shard to destination shard, then config metadata is updated, then source deletes its copy
- Migration causes I/O spike and can impact latency — schedule during maintenance windows or tune `balancerWindow`

### 6.4 Scatter-Gather vs Targeted Queries

```javascript
// Targeted (shard key in query) → mongos routes to 1 shard
db.orders.find({userId: "user123"})   // if sharded on userId

// Scatter-gather (no shard key) → mongos queries ALL shards, merges results
db.orders.find({status: "active"})   // all shards queried
```

Scatter-gather is expensive at scale. Design data model and queries to include shard key.

## 7. Multi-Document ACID Transactions

### 7.1 How They Work (Single Replica Set)

MongoDB 4.0+ supports multi-document transactions within a replica set.

```javascript
const session = client.startSession();
session.startTransaction({
  readConcern: {level: "snapshot"},
  writeConcern: {w: "majority"}
});
try {
  db.accounts.updateOne({_id: "A"}, {$inc: {balance: -100}}, {session});
  db.accounts.updateOne({_id: "B"}, {$inc: {balance: +100}}, {session});
  session.commitTransaction();
} catch (e) {
  session.abortTransaction();
}
```

**Under the hood**: WiredTiger's MVCC provides snapshot isolation within the transaction. All writes are buffered until commit. On commit, changes are applied and oplog entries written atomically.

### 7.2 Transactions in Sharded Clusters (4.2+)

Sharded transactions use **two-phase commit (2PC)** across shards:

1. **Prepare phase**: coordinator (mongos) sends `prepareTransaction` to all participant shards → each shard acquires locks and writes prepare records to its oplog
2. **Commit phase**: coordinator writes commit decision to its local config, then sends `commitTransaction` to all shards

**Performance cost**: sharded transactions are expensive (multiple network round trips, 2PC overhead). Design data model to avoid cross-shard transactions.

### 7.3 Change Streams

Change streams provide real-time ordered change events from collections, databases, or the entire cluster.

```javascript
const changeStream = db.orders.watch([
  {$match: {"operationType": {$in: ["insert", "update"]}}}
]);
changeStream.on("change", change => console.log(change));
```

**How it works**: change streams tail the oplog internally. The `resumeToken` (an oplog timestamp) allows resuming after a disconnect without missing events. Events are ordered within a shard; cross-shard ordering uses a cluster-wide logical timestamp (clusterTime).

**Use cases**: real-time notifications, CDC for downstream systems, cache invalidation, audit logs.

## 8. Data Model and Physical Layout

### 8.1 Conceptual Model

```
Keyspace (≈ database schema)
  └── Table
        └── Partition (unit of distribution)
              └── Row(s) ordered by clustering key
```

```sql
CREATE TABLE time_series (
    device_id   UUID,           -- partition key
    recorded_at TIMESTAMP,      -- clustering key (sort order)
    metric      TEXT,
    value       DOUBLE,
    PRIMARY KEY (device_id, recorded_at)
) WITH CLUSTERING ORDER BY (recorded_at DESC);
```

**Partition key** = `device_id`: determines which node(s) store this partition. All data for one partition is co-located on the same node(s).

**Clustering key** = `recorded_at`: determines sort order within the partition. Physical storage is sorted by clustering key.

**Physical layout on disk**:
```
Partition key: device_id=UUID-1
  Row: recorded_at=T3 → {metric: "cpu", value: 92.1}
  Row: recorded_at=T2 → {metric: "cpu", value: 88.5}
  Row: recorded_at=T1 → {metric: "cpu", value: 75.0}
```

A query `WHERE device_id = X AND recorded_at > T` → single partition lookup → single node → fast.

### 8.2 Mental Model Shift from SQL

| SQL thinking | Cassandra thinking |
|-------------|-------------------|
| Design tables to avoid redundancy | Denormalize; design tables around queries |
| JOINs handle multi-table queries | No JOINs; duplicate data across tables |
| Indexes added after the fact | Query patterns drive schema design from start |
| Transactions across tables | Single-partition atomicity only |
| Flexible queries with WHERE | WHERE must include partition key |

## 9. LSM-Tree Storage

### 9.1 Write Path

```
Client Write
    │
    ▼
CommitLog (disk, sequential, append-only) ← durability guarantee
    │
    ▼
MemTable (in-memory, sorted skip-list, per table per node)
    │
    ▼  (flush: when MemTable reaches threshold, e.g., 64MB)
SSTable (Sorted String Table, immutable, on-disk)
    │
    ▼  (compaction)
Merged SSTables
```

**CommitLog** = Cassandra's WAL. Written before MemTable update. On crash: replay CommitLog to reconstruct MemTable. Shared across all tables (unlike PostgreSQL's per-table WAL).

**MemTable**: in-memory write buffer. One MemTable per table per node, implemented as a sorted structure (ConcurrentSkipListMap in Java implementation). Flushed to SSTable when:
- Size threshold: `memtable_heap_space_in_mb` exceeded
- Flush triggered by commitlog segment size
- Manual `nodetool flush`

### 9.2 SSTable Structure

An SSTable is a set of immutable files per partition key range:

```
data-mc-1-big-Data.db        ← actual row data
data-mc-1-big-Index.db       ← partition key → data file offset
data-mc-1-big-Summary.db     ← sparse index into Index.db (every N keys)
data-mc-1-big-Filter.db      ← Bloom filter (O(1) partition existence check)
data-mc-1-big-Statistics.db  ← table stats, min/max timestamps, tombstone counts
data-mc-1-big-CompressionInfo.db  ← compression chunk offsets
```

**Read path through an SSTable**:
1. Check Bloom filter → partition probably/definitely not here
2. Binary search Summary.db → approximate offset in Index.db
3. Scan Index.db → exact offset in Data.db
4. Read data from Data.db

### 9.3 Compaction Strategies

#### STCS — SizeTieredCompactionStrategy

```
Principle: merge SSTables of similar size together

Level 0: [1MB] [1MB] [2MB] [1MB]  ← 4 similar-sized → merge
Level 1: [4MB]
Level 1: [4MB] [5MB] [4MB] [4MB]  ← 4 similar-sized → merge
Level 2: [17MB]
```

- **Write amplification**: low (2-4x). Best write throughput.
- **Read amplification**: high (must check all SSTables for a key).
- **Space amplification**: ~2x during compaction (temp space needed).
- **Use when**: write-heavy workloads; cold data rarely read; no strict latency SLAs.

#### LCS — LeveledCompactionStrategy

```
L0: [SSTable][SSTable][SSTable]  ← STCS within L0
L1: [10MB max per SSTable, no overlapping key ranges]
L2: [10x L1 = 100MB total, non-overlapping]
L3: [10x L2 = 1GB total, non-overlapping]
```

Key invariant: **within levels L1+, SSTables do not overlap in key range**. A partition key exists in at most one SSTable per level.

- **Read amplification**: O(# levels) — excellent. A read checks at most 1 SSTable per level.
- **Write amplification**: high (10-30x) — each write gets rewritten as levels are leveled.
- **Space amplification**: ~1.1x (minimal).
- **Use when**: read-heavy workloads; strict read latency requirements; random reads dominate.

```
innodb_log_file_size analogy: LCS = B+Tree (read optimized), STCS = LSM default (write optimized)
```

#### TWCS — TimeWindowCompactionStrategy

Designed for **time-series data with TTL**:

```
Window: 1 day
  Day 1 SSTables → compacted into 1 SSTable (then aged as a unit)
  Day 2 SSTables → compacted into 1 SSTable
  Day 3 SSTables → compacted into 1 SSTable

When TTL of Day 1 expires: drop entire Day 1 SSTable file — O(1) deletion, no read needed
```

- **Write amplification**: low (2-4x, similar to STCS within each window).
- **Space amplification**: minimal (expired windows drop as whole files).
- **Read amplification**: depends on window count; bounded by active windows.
- **Requirement**: writes must be monotonically increasing by time (no late data). Out-of-order writes cause windows to never expire.
- **Use when**: IoT sensor data, application logs, event streams with uniform TTL.

```sql
CREATE TABLE events (
    device_id UUID,
    event_time TIMESTAMP,
    data TEXT,
    PRIMARY KEY (device_id, event_time)
) WITH default_time_to_live = 86400  -- 1 day TTL
   AND compaction = {
     'class': 'TimeWindowCompactionStrategy',
     'compaction_window_unit': 'HOURS',
     'compaction_window_size': 1
   };
```

## 10. Consistent Hashing and Data Distribution

### 10.1 Token Ring

Each node owns a range of the token space (0 to 2^64-1 for Murmur3 partitioner). A partition key is hashed to a token → assigned to the first node clockwise that owns that token range.

### 10.2 Virtual Nodes (vnodes)

Without vnodes: each physical node owns one contiguous token range. Adding/removing a node requires large data movement.

With vnodes (default since Cassandra 1.2): each physical node owns **many small token ranges** (`num_tokens = 256` by default) distributed around the ring.

```
Physical Node A → owns tokens: [T1-T10], [T45-T55], [T200-T210], ...
Physical Node B → owns tokens: [T11-T20], [T56-T65], [T211-T220], ...
```

Benefits:
- Adding a node: it takes vnodes from many existing nodes → proportional data movement
- Heterogeneous hardware: nodes with more RAM/disk get more vnodes
- Better load balancing

### 10.3 Replication Factor and Placement

```sql
CREATE KEYSPACE myapp WITH REPLICATION = {
  'class': 'NetworkTopologyStrategy',
  'dc1': 3,
  'dc2': 2
};
```

`NetworkTopologyStrategy`: for each replica, place on the next N nodes in the ring, skipping nodes in the same rack (rack-aware placement).

## 11. Read and Write Path

### 11.1 Write Path

```
Client → Coordinator node (any node, routes by partition key)
Coordinator computes token → identifies replica nodes (based on RF)
Coordinator sends write to replica nodes concurrently
```

Consistency levels for writes:
| CL | Requirement |
|----|-------------|
| `ONE` | 1 replica acknowledges |
| `QUORUM` | Majority of all replicas (`RF/2 + 1`) |
| `LOCAL_QUORUM` | Majority in local DC (multi-DC setups) |
| `ALL` | All replicas acknowledge |
| `ANY` | Even a hinted handoff counts |

Replica write process:
1. Write to CommitLog (sequential disk)
2. Write to MemTable (memory)
3. Acknowledge to coordinator

### 11.2 Read Path

```
Client → Coordinator
Coordinator sends read to replica nodes based on CL
```

**For CL=QUORUM read**:
1. Coordinator contacts `RF/2 + 1` replicas
2. One replica returns full data; others return digest (hash of data)
3. If digests match → return data to client
4. If digests disagree → read repair triggered (fetch full data from all, reconcile)

**Read repair**: reconcile inconsistencies by writing latest version back to stale replicas. Happens in foreground (blocking) or background (`read_repair_chance`).

**Hinted handoff**: if a replica is down during write, coordinator stores a "hint" (intended write). When the replica comes back, coordinator replays hints. Bounded by `max_hint_window_in_ms`.

### 11.3 Consistency Level Tradeoffs

Strong consistency requires: `W + R > RF`

```
RF=3:
  CL=QUORUM write (W=2) + CL=QUORUM read (R=2) → W+R=4 > 3 → strongly consistent
  CL=ONE write (W=1) + CL=ONE read (R=1) → W+R=2 ≤ 3 → eventually consistent (stale reads possible)
  CL=ALL write + CL=ONE read → W+R=4 > 3 → strongly consistent (but write unavailable if any replica down)
```

**LOCAL_QUORUM** is the recommended consistency for multi-DC: strong within DC, async replication cross-DC.

## 12. Tombstones

### 12.1 What Tombstones Are

In Cassandra, **DELETE writes a tombstone** — a special marker with a deletion timestamp — rather than physically removing data. The actual data is removed during compaction when the tombstone is old enough.

```
Write:  {user_id: 1, status: "active"}  → stored in SSTable
Delete: {user_id: 1}                    → tombstone written with local_deletion_time
Read:   SSTable has both → return tombstone (deleted)
Compaction (after gc_grace_seconds): tombstone + old data dropped
```

### 12.2 gc_grace_seconds

Default: 864,000 seconds (10 days). Tombstones are retained for this duration before being eligible for deletion during compaction.

**Why**: replicas may be down. If a tombstone is deleted before a down replica comes back, the replica would resurrect the old data (tombstone "wins" only if it's present; absence means the data wins).

`gc_grace_seconds` must be longer than the maximum time a replica can be down before being considered dead and repaired.

### 12.3 Tombstone Avalanche

A partition with millions of tombstones causes:
1. **Read timeout/failure**: reading the partition must scan all tombstones
2. **GC pressure**: JVM must garbage collect all in-memory tombstone objects
3. **Compaction pressure**: compaction must process all tombstones

**Causes**: deleting many rows from a single partition; TTL expiry for wide partitions.

**Solutions**:
- Use TTL instead of explicit deletes where possible (TTL creates tombstones but they're bounded)
- TWCS for time-series (entire SSTables drop, no per-row tombstones)
- Avoid wide partition anti-pattern
- `tombstone_warn_threshold` / `tombstone_failure_threshold` to catch the problem early

## 13. Secondary Indexes

### 13.1 Local Secondary Indexes (built-in, avoid)

Cassandra's original secondary indexes are **local** — each node indexes only the data it owns.

**Problem**: querying by a secondary index requires **scatter-gather** (query all nodes), then filter locally. With RF=3, each row is on 3 nodes, causing triplicate work.

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
-- Without secondary index: partition key required
-- With secondary index on email: works, but queries ALL nodes
```

**Anti-pattern**: don't use built-in secondary indexes on high-cardinality columns or frequently-queried non-primary-key columns in large clusters.

### 13.2 SASI (SSTable Attached Secondary Index)

Introduced in Cassandra 2.1. Index is stored alongside SSTable, not in a separate table. Supports:
- Prefix matching (`LIKE 'prefix%'`)
- Range queries on non-primary-key columns
- Faster than local secondary indexes for some workloads

**Limitations**: still local (per-node), not recommended for production in high-cardinality scenarios.

### 13.3 SAI (Storage Attached Index) — Cassandra 4.x

SAI is the modern replacement, introduced as GA in Cassandra 4.0:
- Per-column, attached to SSTables (no separate table)
- Supports both equality and range queries
- Much better read performance than SASI
- Still local; scatter-gather for non-partition-key queries, but optimized

For truly efficient non-partition-key queries: use a separate table with the query field as partition key (denormalization), or use a search engine like Elasticsearch/Solr with Cassandra.

## 14. Anti-Patterns

| Anti-Pattern | Problem | Solution |
|-------------|---------|----------|
| Large partitions (>100MB) | Slow reads, GC pressure, compaction stalls | Add a time bucket or sub-partition to partition key |
| `ALLOW FILTERING` | Full cluster scan; ignores all indexes | Redesign schema or create proper index |
| Unbounded `IN` clauses | High coordinator work; many separate reads internally | Batch client-side or redesign |
| Using secondary index on high-cardinality column | Scatter-gather on all nodes | Separate table with column as partition key |
| Monotonically increasing partition key | One partition grows unbounded (hot partition) | Bucket by time period |
| Cross-partition transactions | Cassandra has no 2PC across partitions | Use lightweight transactions (LWT) within a single partition only |

### ALLOW FILTERING — Why It's Dangerous

```sql
-- This query requires Cassandra to scan every single partition across all nodes:
SELECT * FROM orders WHERE amount > 1000 ALLOW FILTERING;
-- On a 1TB table: full cluster scan, potentially millions of reads
```

The `ALLOW FILTERING` keyword is a red flag in schema review. Its presence means the query cannot use the partition key to route efficiently.

---

## 15. CQL vs SQL Mental Model

| Concept | SQL | CQL |
|---------|-----|-----|
| Unique row identifier | Primary key (any column) | Partition key (routing) + clustering key (ordering) |
| Query flexibility | Any column in WHERE | Must include partition key |
| Consistency | ACID, serializable | Tunable per operation |
| JOINs | Yes | No |
| Transactions | Multi-table | Single partition (LWT for conditional writes) |
| Schema changes | ALTER TABLE (usually fast) | ALTER TABLE (DDL is instant; data transformation separate) |
| NULL | Stored as null | Not stored (tombstone written on explicit set to null) |

## Interview Quick-Fire Reference

### MongoDB

| Question | Key Answer |
|----------|-----------|
| WiredTiger vs old MMAPv1 | B+Tree indexes; document-level concurrency; compression; MVCC snapshots |
| Why avoid monotonic shard key? | All new writes go to last chunk; hotspot; unbalanced shards |
| How does oplog work? | Capped collection recording all writes; secondaries tail it |
| $lookup implementation | Hash join (default) or index NL join; must avoid cross-shard $lookup |
| Multikey index limitation | Only one array field per compound index |
| Change streams mechanism | Oplog tailing; resumeToken for resumability |

### Cassandra

| Question | Key Answer |
|----------|-----------|
| Why STCS vs LCS? | STCS: write-heavy; LCS: read-heavy with range queries |
| What is a tombstone danger? | Tombstone avalanche: millions of tombstones in a partition → read timeouts |
| W+R > RF means what? | Strong consistency; QUORUM+QUORUM for RF=3 = strongly consistent |
| Why are local secondary indexes problematic? | Scatter-gather across all nodes for every query |
| What is gc_grace_seconds? | Time to retain tombstones so down replicas don't resurrect deleted data |
| TWCS requirement | Monotonically increasing writes only; no late data; all rows must have TTL |
| ALLOW FILTERING danger | Forces full cluster scan; bypasses all partitioning benefits |
